## 0. Thesis (the one idea to hold in your head)

The scaffold already has the right spine: a **pure, immutable, discrete sim core**
(`state -> state`), a **single rigid snake under gravity**, and **two verbs**
(`move`, `strike`). Almost the entire ability toolbox is expressible by changing *one*
thing: the closed `CellType` union becomes a small **tagged entity with orthogonal
property flags**. Walls, fruit, exits, and most future abilities (anchors, heat, doors,
carried items, decoys) are then *data*, not new code paths.

The honest framing of that win (see §2.3, D6): the payoff is that the **existing rule
call-sites (step / support / eat / win) stop needing per-ability branches** because they
read flags. A *new* entity is not free — it costs one `PRESETS` row + one renderer `kind`
branch + (for active behaviours) one resolution pass or verb. What the flag-set buys is
that the hot rule functions never grow a `switch` per ability; that is the simplicity win,
not "abilities become free."

Two abilities genuinely demand a new structural concept and nothing else does:
**per-segment grip** (anchoring, World 3) and **a second snake** (co-op, last and
cut-first). Each gets a spike before content.

That is the whole design. The rest is content and verification. The single rule that
makes it all work — and that every ability preserves — is:

> **The rules are pure functions `state -> state`; the test suite IS the spec.**
> Determinism + immutability give undo-as-snapshot-stack, referential-equality no-op
> detection, and unit tests that read like a truth table. Every ability is admitted to
> the model **only** with an oracle that pins its behavior first.

## 1. Scope

### 1.1 In scope (this HLD designs the whole game)

- A **core data-model evolution** expressing the full toolbox with the fewest concepts,
  each extension pinned to the world that first needs it (and *not before*), recorded in a
  reviewer-checkable "no on-ramps" ledger (§2.3).
- The **complete ability toolbox**: body-as-structure, strike/coil, length/growth,
  wrap (emergent body-blocking), rattle (redesigned, agent-free), heat-sense/dark,
  swallow & carry, shed-skin decoy, co-op/relay.
- The **world curriculum** (8 worlds), sequenced into independently shippable increments.
- The **scoring model**: eggs (solve / hidden / constraint) + move-par, kept optional,
  non-blocking, and entirely outside the pure core.
- The **renderer/animation** plan: clear-and-rebuild is the firm, permanent decision; the
  single isolated place a persistent-mesh pivot *might* later be justified is a one-line
  note, not a committed module.
- The **almanac ship plan** (night-cab style, Vite `base:"./"`).
- The **acceptance oracles** (property battery + per-mechanic directed tests) that define
  "done" for each increment.
- **Authoring ergonomics from Inc 1**: a tiny ASCII-art room format with a pure, tested
  parser, and a "record the current play session as par" tool in the shell (§2.6, §6 D23).

### 1.2 Out of scope (explicitly cut or deferred — see Decision Log)

- **Touch / gamepad / on-screen controls / accessibility / key remap.** Desktop keyboard
  + mouse-orbit only, as today. (D-OUT-1)
- **JSON level format + in-browser editor.** Levels stay TS literals (authored via the
  ASCII room format, §2.6). `LevelDef` is already plain serializable data, so promotion to

  outgrows the TS-literal-plus-ASCII path. (D-OUT-2)

  Re-evaluate only if a world's puzzle genuinely requires variable reach. (D-OUT-3)
- **A creature AI / pathfinding / sound-propagation system.** The "Rattle" world is
  redesigned around a deterministic, snake-triggered plate->gate hazard (no autonomous
  agent). (D-OUT-4)
- **Full multi-snake co-op as a separate game mode.** Folded into a minimal "second body"
  model and flagged as the first thing to cut if production runs long. (D-OUT-5)
- **Formal constriction / crushing** (a `surrounds` predicate + a `crushable` entity).

- **Server-side anything, audio engine, timers, lives, accounts, leaderboards,
  telemetry.** "Rattle" ships silent with a visual pulse; real sound FX are a later
  cosmetic pass at most.
- **Progress persistence beyond one `localStorage` key** (`coil.progress.v1`: highest
  world unlocked + per-room egg flags). No cloud sync, no migration (sandbox).

## 2. The design

### 2.1 Core data model — the one unifying change

**Today** (`types.ts`):

```ts
export type CellType = "wall" | "fruit" | "exit";   // closed union
cells: Map<string, CellType>;                        // one tag per cell
```

The closed union forces every new ability into either a new string variant *and* matching
branches in `tryStep`/`settle`/`checkWin`, or a parallel structure. That is the "many
special cases" trap.

**The change** — replace the string tag with a tiny **entity record of orthogonal
properties**. The properties are exactly the questions the rules already ask:

```ts
export interface Entity {
  kind: EntityKind;     // for the renderer + readability; rules read flags, not kind
  solid?: boolean;      // blocks a step (today: wall)
  supports?: boolean;   // bears the snake's weight from below (today: wall)
  eat?: boolean;        // stepping onto it grows the snake & removes it (today: fruit)
  win?: boolean;        // head on it -> won (today: exit)
  // --- added only when its world arrives (the §2.3 ledger says when) ---
  grip?: boolean;       // snake may anchor a segment here          (Inc 2 / World 3)
  heat?: boolean;       // renderer-only: visible in the dark        (Inc 3 / World 5)
  trigger?: string;     // a plate; pressing it fires a mechanism id (Inc 3 / World 4)
  door?: string;        // a solid this trigger id toggles open      (Inc 3 / World 4)
  pickup?: boolean;     // swallowable; stored on a segment as carry (Inc 4 / World 6)
  egg?: boolean;        // a collectible scoring marker (no growth)  (Inc 4 / scoring)
}
export type EntityKind =
  | "wall" | "fruit" | "exit"
  | "anchor" | "heatlamp" | "plate" | "gate" | "object" | "egg";  // grows with increments

cells: Map<string, Entity>;
```

**Why a flag-set and not the closed union (the contested call, resolved):** one judge
preferred keeping a closed `CellType` union with a `cellProps()` lookup, to forbid
nonsensical cells (`win+solid+eat`). We adopt the flag-set as the spine because it is the
single primitive that keeps the **hot rule call-sites branch-free** (they read flags, not
a per-ability `switch`), and we **graft the invalid-combination guard**: a small frozen
`PRESETS` table is the *only* sanctioned way to construct an entity, so authors pick a
named preset, never raw flags. This gives the flag model's extensibility without flag-soup
invalidity. (Honest framing: a *new* entity still costs a `PRESETS` row + a renderer `kind`
branch + any active resolution pass — see §2.3 and D6/D10.)

```ts
const PRESETS = {
  wall:  { kind: "wall",  solid: true, supports: true },
  fruit: { kind: "fruit", eat: true },
  exit:  { kind: "exit",  win: true },
  // added by the increment that needs them:
  anchor: { kind: "anchor", solid: true, supports: true, grip: true },   // Inc 2
  plate:  { kind: "plate", trigger: /*id set per cell*/ "" },            // Inc 3
  gate:   { kind: "gate",  solid: true, door: "" },                      // Inc 3
  // ...
} as const;
```

- The three existing types collapse to presets *with no rules change to their meaning*.
- The rules read **flags, never the kind string**: `tryStep` blocks on `e.solid`;
  `isSupported` checks `e.supports`; eating checks `e.eat`; `checkWin` checks `e.win`.
  This is a *mechanical refactor* of the existing four call-sites — same logic, one
  indirection.
- **Construct only via PRESETS — structurally near-enforced, not mere convention (F6).**
  Both `LevelDef` and the ASCII parser reference presets **BY NAME only** (a preset id +
  optional per-cell overrides such as a `trigger`/`door` id); **raw `Entity` object
  literals never appear in level data.** Because there is no syntactic path to hand-write a
  flag-soup entity in a level, "construct only via PRESETS" is structurally near-enforced.
  The remaining gap (a row in `PRESETS` itself being malformed) is closed by the build-time
  `T-PRESETS` assert (§4.0/§4.1).
- **PRESETS single-pass construction guard (M1, §2.4/§2.2.6).** Any preset whose
  *solidity is DERIVED during `applyMechanisms`* (today: `gate`, whose `solid` is
  recomputed each turn) MUST be constructed with `supports: false` and MUST NEVER set
  `supports`. This is a frozen-table construction invariant (a build-time assert over
  `PRESETS`, part of `T-PRESETS`): because no mechanism-derived entity can ever be a
  grounding source, a mechanism state change can never alter the grounded set, so settle
  never needs to re-run after `applyMechanisms` — the settle<->mechanism pipeline is correct
  in a single pass (see §2.4).
- **Inc-1 ordering of the refactor (load-bearing).** The §4.0 property battery and the
  §4.1 oracle-gap tests are landed GREEN on the **pre-refactor** code first; the entity
  refactor is then done against that hardened net. The *existing + new* tests assert
  behavior, not representation, so they must stay green across the refactor — and because
  the net was proven green on the old code, a post-refactor failure is unambiguously the
  refactor's fault. This is the Inc-1 acceptance gate. The Inc-1 battery is driven over a
  **committed fixed adversarial corpus** (no fast-check; see §4.0). The pure-tail
  equivalence oracle T-RESOLVE-EQUIV is **not** an Inc-1 artifact — it lands at Inc 3 with
  the `resolve` tail extraction, and there it becomes a fast-check-driven differential
  oracle against a **frozen pre-extraction comparand** (§4.3, D21).
- **No flag is added until the increment that uses it.** `grip`/`heat`/`trigger`/`door`/
  `pickup`/`egg` are listed above for foresight but are absent from the Inc-1 type and
  presets. The foresight lives in the *naming convention* (orthogonal booleans), not in
  unused code — see the ledger (§2.3).

This single change carries anchors, heat lamps, pressure plates + gates, swallowable
items, and decoys *without growing the hot rule functions*: the existing call-sites read
flags and never branch per ability. New entities still cost their PRESETS row + renderer
branch + any active pass (§2.3).

#### State shape (final, Inc-4)

```ts
export interface GameState {
  snake: Segment[];     // ordered, head first  (was Vec[])
  cells: Map<string, Entity>;
  strikeRange: number;
  floorY: number;
  status: Status;       // "play" | "won" | "dead"
  name: string;
  // added only by the increment that needs it:
  triggers?: Set<string>;    // currently-pressed trigger ids (Inc 3)
  bodies?: Segment[][];      // additional snakes for co-op   (Inc 4, may be cut)
}

// Inc 1: Segment === Vec (a bare {x,y}); the alias keeps Inc-1 byte-identical.
// Inc 2 widens it:
export interface Segment { x: number; y: number; anchored?: boolean; carry?: Entity; }
```

> **Migration note (sandbox: no back-compat owed).** `snake: Vec[]` becomes
> `snake: Segment[]` with `Segment` a superset of `Vec`, so Inc-1 code is unchanged.
> `LevelDef.cells` becomes a list of `{x, y, preset, ...overrides}` (**preset BY NAME**,
> never a raw entity literal — F6); `buildState` builds the entity map from presets. The
> whole change is type-additive. We do **not** rename `snake` to a `snakes[]` array up front
> (a runner-up did, then listed it as its own residual risk — an on-ramp to nowhere); co-op
> widens the container last, behind a golden-trace equivalence oracle (§2.2.10, §4.4).
>
> **Bodies-absent invariant (M2, load-bearing for the migration oracle).** Single-snake
> states keep `bodies` **ABSENT (`undefined`), never `[]`.** The kernel iterates
> `[snake, ...(bodies ?? [])]` and **never writes `bodies` back** on a single-snake state.
> This is what makes T-COOP-MIGRATE *byte-identity* attainable (§2.2.10, §4.4): a co-op
> kernel replaying a single-snake trace produces snapshots with `bodies` still `undefined`,
> deep-equal to the Inc-0 originals.

### 2.2 How every ability is expressed

The toolbox, each mapped to the **smallest** model addition. "New verb?" tracks the
control-surface budget: the design strongly prefers **zero new verbs** (only `move` and
`strike` exist) and expresses abilities as *board state the existing verbs interact with*.

| Ability | Model addition | New verb? | Increment |
|---|---|---|---|
| Body-as-structure | none (emergent from snake occupancy + rigid fall) | no | 0 |
| Strike / coil | none (`strike` verb already present) | no | 0 |
| Length / growth | none (`eat` flag already present) | no | 0 |
| Wrap (body-blocking) | none — **emergent** from existing body occupancy (no `surrounds` predicate; D29) | no | (emergent, any world) |
| Anchor (was "climb") | `Entity.grip` + `Segment.anchored` + one toggle | **1 verb: anchor** | 2 |
| Rattle (redesigned) | `Entity.trigger`/`door` + a mechanism pass | no | 3 |
| Heat-sense (dark) | `Entity.heat` (renderer-only) | no | 3 |
| Swallow & carry | `Segment.carry` + eat-of-`pickup` entity | no (deposit modifier) | 4 |
| Shed-skin decoy | deposit a body-shaped `solid` from carry | reuses carry | 4 |
| Co-op / relay | `bodies: Segment[][]` (minimal) | no (Tab cycles) | 4 (cut-first) |

> **Wrap note (D29).** Wrap/constrict stays in the ability toolbox as **emergent
> body-blocking**: you can surround a target with your own body so existing occupancy
> blocks/traps it — this needs **no model addition** (no formal `surrounds` predicate, no
> `crushable` entity; both cut, see §2.2.5 / D29). If a future world genuinely needs formal
> *crushing*, define the predicate then against that concrete room.

The conscious control-surface budget for the entire game is **two movement verbs
(`move`, `strike`) + one toggle (`anchor`) + a deposit modifier + Tab (co-op only)**.

#### 2.2.1 Body-as-structure (W1, exists)
No model. The snake's own segments already block steps (`tryStep` body check) and occupy
space; a coiled body is a wall you made. Puzzles use this directly: park your tail to
stand on later, or to block a fall, or to bridge a gap with the C-shape (tail-vacate).

#### 2.2.2 Strike / coil (W1, exists)
No model. `strike(dir)` launches the head up to `strikeRange` with gravity suppressed,
then one settle. **Resolved open questions (now acceptance oracles, §4):**
- *Strike ending over void:* the final `settle()` runs normally — a strike that lands
  over a gap **falls and may die**. Intended (it's the risk that makes strike a
  decision). Oracle T-STRIKE-3.
- *Strike over fruit mid-flight:* the head **eats and grows mid-strike** (a `tryStep` onto
  `eat` keeps the tail). Intended; it lets a strike both cross a gap and grab a fruit.
  Oracle T-STRIKE-4.
- *Win mid-strike:* if the head crosses an exit cell during the strike flight, the game is
  won **at that instant** — `checkWin` runs on each strike flight step (§2.3, §2.4).
  Oracle T-WIN-STRIKE-CROSS (§4.1). **A win, once set, is terminal:** if a strike crosses
  an exit mid-flight and the remaining flight + final settle would otherwise end over the
  void, the status stays `won`, not `dead` — see status-monotonicity (§2.5, D28) and
  oracle T-WIN-STRIKE-VOID (§4.1).
- **Status-monotonicity note (single-body).** The per-step `checkWin` during strike
  flight decides the *single-body* win. In multi-body play (Inc 4) the per-step early-win
  does **not** fire while `bodies` is present, because a multi-body win requires ALL heads
  on exits and `checkEnd` is the win authority (§2.5).

#### 2.2.3 Length / growth (W1-W2, exists)
No model. `eat` entities grow the snake (+1, tail kept). Growth's puzzle role: longer =
reaches farther *as a structure* and fills more cells (easier to trap yourself).
**Coil-to-charge is cut (D-OUT-3)** — length does not change `strikeRange`. Reach-from-
length is already expressible: a longer body physically spans more cells.

#### 2.2.4 Anchor (W3 "The Climb") — the first real structural addition
The only sub-rigid-body concept the game needs. An `anchor` entity carries `grip:true`. A
new verb:

```ts
anchor(s): GameState     // toggle: anchor/release the segment currently on a grip cell
```

**Anchored grounding is DERIVED FROM GRIP each turn — there is no latch (D25, mirrors the
gate fix D19).** The stored `Segment.anchored` flag is *intent*; the grip cell is the
*truth*. `settle()` treats a segment as a grounding source **only when it is
`anchored` AND currently sitting on a grip cell**:

```
groundingSource(seg) := (world cell under/at seg has supports) OR
                        (seg.anchored AND cell at seg has grip)
```

When such a segment exists, **`settle()` does not fall** (the snake hangs). This folds
into the transitive-grounding settle (§2.5): an anchored-on-grip segment is a grounding
source, exactly like a world `supports` cell. Climbing is then `anchor -> move head up
along a grip wall -> re-anchor -> release tail`. This expresses **perching/gripping**
without a per-cell grip graph or rope physics.

**Consequence (the thing the latch would have hidden):** a segment carried *off* its grip
cell stops grounding the snake. If you move the body so the anchored segment leaves the
grip cell, it is no longer a grounding source and (absent any other source) **the snake
falls** — the `anchored` flag alone cannot levitate it. Oracle T-ANCHOR-CARRY (§4.2).

We explicitly **reject per-segment partial support** (a snake half-hanging off a ledge
sagging under its own weight). That is the un-holdable full-3D knot the design forbids;
anchoring is binary (you grip or you don't), keeping `settle` a short function forever.

> **Why a verb here and nowhere else:** anchoring is a *choice the player makes about
> their own body* that gravity must read — it cannot be board data alone. Every other
> ability is board data the existing verbs touch, so they cost no verb. Two movement verbs
> plus one toggle is the whole control surface. Hold-in-head test: passes.
>
> **Control-surface key-sharing caveat (F8, §2.2.8/§2.2.10/D8).** The `anchor` toggle and
> the carry `deposit` modifier share a key on the assumption they never coexist in one
> room. To keep that assumption honest, **any room (the W8 Finale included) must EITHER
> avoid combining anchor + carry in the same room OR the shipped build assigns them
> distinct keys.**

**Spike before content (R14 graft):** before authoring World 3, build one throwaway room
proving a 3-cell anchored snake holds while the head climbs/strikes. If the feel is
wrong, only the spike is lost, not a world.

#### 2.2.5 Wrap (W2 / emergent) — body-blocking, not a rule
**No model addition.** Wrapping a target is **emergent body occupancy**: surround a target
with your own body and the existing segment-occupancy/`tryStep` rules already block it from
moving — you have trapped it with the body you made (cf. §2.2.1). There is **no formal
`surrounds` predicate and no `crushable` entity** — both are **cut (D29)**.

**Why cut:** no shipped world needs formal *crushing*. The W2 "Growth" twist as specified —
"a room you can only solve if you don't over-grow" — is a length-as-trap puzzle, not a
constriction puzzle. A formal `surrounds` predicate over an unbounded sparse map carried a
genuine border-definition problem (what counts as "escaped" when there is no grid edge);
rather than ship a predicate no room exercises, we delete it. **If a future world genuinely
needs formal crushing**, define the `surrounds` predicate **then**, against that concrete
room (so its border definition is grounded in a real requirement, not speculation). This
supersedes the V3 bbox-flood-fill fix — the predicate is gone, so its border-definition
problem is moot.

#### 2.2.6 Rattle (W4 "Pressure"), redesigned to kill the AI dependency
The original "rattle = sound triggers a creature" needs a creature/AI/sound-propagation
model — three systems, none of which fit "hold it in your head," and the highest-scope
thing the runner-up drafts kept building. **Redesign:** keep the *theme* (your body's
motion triggers the environment) but make it **deterministic and agent-free**:
- A `plate` entity has `trigger:"g1"`; a `gate` entity has `door:"g1"` and `solid:true`.
- After settle, a **mechanism pass** computes
  `triggers = { ids of plates currently under a snake segment }`, then derives **every
  turn, from scratch**:

  ```
  gate.solid = !(gate.door in triggers) && !(gate cell occupied by any segment of any body)
  ```

  A gate is **held open by EITHER its plate OR a body in its mouth**; its solidity is
  *derived each turn from current triggers + occupancy, with NO cross-turn latched state*.
  Pure, deterministic, undoable. **Occupancy is an existential set-membership test**
  ("occupied by ANY segment of ANY body") and is therefore **order-independent across
  bodies**.
- **Single-pass invariant (M1, construction guard).** A `gate`'s solidity is *derived* in
  `applyMechanisms`, so by the PRESETS construction guard (§2.1, §2.4) a `gate` **NEVER
  sets `supports`** (`supports:false`, frozen). Because no mechanism-derived entity is ever
  a grounding source, opening or closing a gate **cannot change the grounded set**, so
  settle never needs to re-run after `applyMechanisms`. This is what keeps the
  `settle -> applyMechanisms -> checkEnd` tail correct in a *single pass* (§2.4).
- "Rattle" flavor: stepping on a plate opens a distant gate; the puzzle is holding a plate
  down *with part of your body* while reaching the gate with the rest — body-as-structure
  again, with a visual pulse for the "rattle."

This is one pure function (`applyMechanisms(s): s`) in the shared `resolve` tail. No
autonomous entities ever. (D-OUT-4.) **Gate solidity is purely derived (no latch):** when
the plate releases AND the gate mouth is empty, the gate re-closes (oracle T-MECH-1); when
the gate mouth is occupied by any segment, the gate stays open and there is **no crush**
(oracle T-MECH-2). There is no triggers-only formula — occupancy is always part of the
derivation.

#### 2.2.7 Heat-sense / Dark (W5) — renderer-only, zero rules
`Entity.heat:true` marks lamps/warm cells. The **core does not change at all** — `heat` is
read solely by the renderer, which dims everything except heat sources, the snake, and a
small radius. All collision/gravity/win rules are identical (regression oracle: every
prior core test passes byte-identically with heat cells sprinkled in). The "dark" puzzle
is purely about *information*: remember geometry you saw lit. Making fog a rule would add
hidden state and wreck testability for zero puzzle benefit. The cheapest world: content +
a renderer lighting mode (still one renderer `kind`/flag branch — see the honest framing in
§2.3).

#### 2.2.8 Swallow & carry (W6 "The Gullet")
`Segment.carry?: Entity`. A `pickup` entity (flag `pickup:true`, removed from `cells` when
stepped on) is stored on the head segment as `carry` instead of growing length. Carrying:
- moves with the segment (it's part of the snake record),
- can be **deposited** via a direction + a deposit modifier into an adjacent empty cell
  (drop a key onto a plate, a block to stand on),
- a carried `solid` block, once deposited, is a normal `cells` entity again.

This expresses "swallow a thing, carry it through, regurgitate it where needed." No new
verb beyond the deposit modifier. **The deposit modifier piggybacks on the anchor key on
the assumption anchor + carry never coexist in one room (D8/§2.2.4).** To keep that honest:
a room must EITHER not combine anchor + carry OR the shipped build gives them distinct keys
(F8 caveat). **Spike carry before authoring W6.**

#### 2.2.9 Shed-skin decoy (W6 twist)
A decoy is just **deposit of a body-shaped `solid` entity from carry**: you "shed" a held
block that then acts as static structure (a fake body / a plate-holder / a step). No new
mechanism — it is carry (§2.2.8) reskinned. **Matter-conservation oracle:** body length +
deposited matter is constant across a shed (§4.4). Cut candidate if W6 is already full.

#### 2.2.10 Co-op / relay (W7 "Two Bodies") — the one genuine fork, minimized
A second snake is the **only** ability the single-snake state cannot fake. Minimal model:
`bodies?: Segment[][]` (the *other* snakes; `snake` stays the active one). Turn model: the
player **controls one body at a time**; `Tab` switches the active body; gravity/mechanism
passes run over *all* bodies each turn. **Inter-body support is the multi-body case of the
transitive-grounding settle (§2.5):** a body is supported iff it transitively rests —
through a chain of bodies it sits on — on a world `supports` cell or an anchored-on-grip
segment. External bodies bear weight; **two bodies resting only on each other with no
ground/anchor BOTH fall** (no mutual mid-air self-support). The settle is a simultaneous
fixpoint and is **order-independent across bodies by construction.**

**Co-op fixpoint precedence — WIN before death (D28, F-coop).** At the fixpoint tail of a
multi-body turn, **WIN is evaluated BEFORE death**: if all heads are on their exits the
state is **`won`**, and that takes precedence over a body reaching the void in the **SAME**
fixpoint (**win-wins-over-death**). Single-body instant-win (per-step `checkWin` during
fall/flight) and status-monotonicity are **unchanged**. **Win = all heads on (their) exits,
decided by `checkEnd`** (the per-step early-win does NOT fire while `bodies` is present —
§2.5, D28); death = any body fully in the void *only when no all-heads win occurred this
fixpoint*. Each verb operates on the active body only. Oracle T-COOP-WIN-PRECEDENCE (§4.4).

**Bodies-absent invariant (M2).** Single-snake states keep `bodies` **absent
(`undefined`), never `[]`**; the kernel iterates `[snake, ...(bodies ?? [])]` and **never
writes `bodies` back** on a single-snake state. This is the precondition that makes
T-COOP-MIGRATE byte-identity attainable.

**Key-sharing caveat (F8/D8).** Because `anchor` and the carry `deposit` modifier share a
key on the no-coexist assumption, the Finale (W7/W8) — like any room — must EITHER avoid
combining anchor + carry OR the shipped build assigns them distinct keys.

This is ~30 lines: `settle`/`isSupported`/mechanism pass iterate `[snake, ...(bodies ??
[])]`. It is the only change that touches the *root* of the state tree, so it is done
**last** and guarded by a **golden-trace migration oracle** (graft from draft-testability,
§4.9): an Inc-0 single-snake golden trace, replayed through the refactored kernel with
`bodies` absent / active-snake-only, must produce **byte-identical** snapshots — proving
the wrap-the-kernel refactor is behaviour-preserving. **Spike co-op** (B strikes off A's
back to an exit) on one throwaway room before authoring W7.

**It is the first thing to cut** if production time is short (D-OUT-5) — W7 can be replaced
by an extra finale-style recombination world using only W1-W6 abilities, with no model
change. Recommendation: **build W1-W6 + Finale first; treat W7 as a stretch world.**

### 2.4 Turn resolution (unified pipeline)

Every verb resolves through the **same** pipeline: the current pipeline plus a shared tail
extracted at Inc 3 (the first increment with more than one post-action recompute). Each
added pass no-ops until its world exists, so Inc 0/1/2 keep their exact behavior.

```
verb(state, dir) ->
  if status != "play": return state (same ref)            [Inc0]
  apply verb's grid steps via tryStep (reads e.solid/e.eat) [Inc0, flag-driven]
    -- checkWin runs after the verb's grid steps           [Inc0]
    -- (strike) checkWin runs on EACH strike flight step    [Inc0]
  if nothing changed: return state (same ref)              [Inc0, no-op identity]
  return resolve(candidate)

resolve(s) ->
  if s.status != "play": return s (same ref)               [STATUS-MONOTONICITY, D28]
  s = settle(s)            (transitive grounding + anchored-on-grip) [Inc0; +anchor Inc2]
    -- settle early-returns if s.status !== "play"          [D28]
    -- checkWin runs on EACH fall step inside settle (single-body) [Inc0]
  s = applyMechanisms(s)   (plates + occupancy -> gates)     [Inc3; no-op if no triggers]
    -- no-op if s.status !== "play"                          [D28]
  return checkEnd(s)       (REDUNDANT FINAL guard, AND the win authority in multi-body;
                            WIN evaluated BEFORE death — D28) [Inc0; multi-head Inc4]

// STATUS-MONOTONICITY (D28): once status leaves "play" (won or dead), settle/
// applyMechanisms/checkEnd are NO-OPS and never downgrade or overwrite it. A "won" snake
// is terminal and does NOT fall.
// Win-during-fall and win-during-strike-flight are RETAINED for the SINGLE body: checkWin
// runs (a) after the verb's grid steps, (b) on each fall step inside settle, and (c) on
// each strike flight step. checkEnd is a final redundant guard AND, in multi-body play,
// the SOLE win authority (the per-step early-win does NOT fire while bodies is present).
// CO-OP PRECEDENCE (D28): in multi-body checkEnd, WIN is evaluated BEFORE death --
//   all heads on exits => 'won', taking precedence over a body in the void the SAME turn.
// Inc4 (co-op): every step maps over [snake, ...(bodies ?? [])]; active snake acts, others
//               are external obstacles/support; win = all bodies on exits (checkEnd).
```

**Single-pass settle<->mechanism invariant (M1).** `applyMechanisms` runs *after* settle
and is never followed by a re-settle. This is correct because of the PRESETS construction
guard (§2.1, §2.2.6): **any entity whose solidity is derived in `applyMechanisms` (e.g. a
`gate`) MUST NEVER set `supports`.** A mechanism state change therefore can never alter the
grounded set, so no re-settle is ever needed. The guard is a frozen-`PRESETS` build-time
assert (part of `T-PRESETS`), not a runtime check.

Purity is preserved end-to-end (every pass is `s -> s`), so **undo stays a snapshot stack
and no-op detection stays referential** (`next === state`). This is load-bearing and
non-negotiable.

> **Win-check semantics (corrected, with monotonicity).** It is **not** true that
> `resolve == settle(checkWin(s))` with a single final win check. For the **single body**,
> win is detected at the **instant** it occurs: a head that falls through an exit cell
> mid-settle, or crosses an exit mid-strike, wins then and there. **Once `won` is set it is
> terminal** — the rest of the resolve tail no-ops on it (D28), so a win never reverts to
> `dead` even if the remaining flight/fall would have ended over void.
> In **multi-body** play, `checkEnd` is the win authority (all heads on exits); the
> per-step early-win does **not** fire while `bodies` is present, so single-step
> monotonicity and the all-heads multi-head rule stay consistent. **Within multi-body
> `checkEnd`, WIN is evaluated BEFORE death** (win-wins-over-death, D28): all heads on
> exits resolves to `won` even if another body reaches the void in the same fixpoint.
> `checkEnd` at the tail is a redundant final guard for the single body and the sole win
> check for multi-body. Oracles T-WIN-FALL, T-WIN-STRIKE-CROSS, T-WIN-STRIKE-VOID, and
> T-COOP-WIN-PRECEDENCE pin this (§4.1, §4.4).

### 2.5 Gravity / support — transitive grounding as a simultaneous fixpoint

Settle is defined **generally and once**, covering single- and multi-body identically.
**Status-monotonicity early-return (D28):** `settle` (and the whole resolve tail) returns
its input **unchanged (same reference)** if `status !== "play"` — a `won` (or `dead`)
snake is terminal and does not fall. A segment/body is **supported** iff it **transitively
rests** — through a chain of bodies it sits on — on a world `supports` cell OR an
**anchored-on-grip** segment. This is a *closure*, not a one-step neighbour check.

> **Inc-1 settle scoping (load-bearing, D30).** This section is the eventual **SPEC**
> (single- and multi-body, anchored-on-grip, transitive closure). **Inc 1 does NOT build
> the fixpoint / body-graph machinery.** What Inc 1 ships is the existing **single-body
> support loop + the status-monotonicity early-return (D28), BYTE-EQUIVALENT to today**
> (the slice's settle, unchanged in behaviour). The pieces of this general spec land with
> the worlds that need them: **anchored-on-grip grounding lands at Inc 2** (with the anchor
> verb), and **multi-body simultaneous iteration lands at Inc 4** (with `bodies[]`). Do NOT
> build the general fixpoint or body-graph at Inc 1 — that is an on-ramp to nowhere. Keep
> this section as the spec, but the fixpoint is **not** attributed to Inc 1 code. (D30)

**Algorithm (order-independent by construction) — the general spec, staged per D30.**
0. If `status !== "play"`, return the state unchanged (same ref). [D28]
1. Compute the **grounded set** as the transitive closure from the grounding sources
   (world `supports` cells + segments that are **`anchored` AND currently on a `grip`
   cell**, D25 — *the anchored-on-grip source lands Inc 2*): a body is grounded if it sits
   directly on a grounding source, or sits on another body that is (transitively) grounded
   (*the multi-body chain lands Inc 4*).
2. Every body **not** in the grounded set drops by 1 cell **SIMULTANEOUSLY**.
3. Repeat to a **fixpoint** (no body is ungrounded, or it has crossed `floorY`).

- **Single-body settle is the degenerate case** of this one fixpoint definition (the
  closure has one body; it is grounded iff some segment sits on a world `supports` cell or
  is anchored-on-grip). **At Inc 1 this degenerate case IS the shipped settle — the
  existing single-body support loop, byte-equivalent to today (D30).**
- **Anchored grounding is derived from grip each turn — no latch (D25).** A segment is a
  grounding source **only while it currently sits on a grip cell**; the stored `anchored`
  flag is intent, the grip cell is the truth. A segment carried off its grip cell stops
  grounding, and (absent another source) the snake falls — oracle T-ANCHOR-CARRY (§4.2).
  This mirrors the gate derived-each-turn fix (D19). *(Lands Inc 2.)*
- **Consequence (the bug this fixes):** two bodies resting only on **each other** with no
  ground/anchor are **not** in the grounded closure, so **BOTH fall**. There is no circular
  mid-air self-support, and the result does not depend on which body is processed first
  (F3 and F17 were the same defect — order-dependence and circular support — both removed by
  the closure definition). *(Multi-body; lands Inc 4.)*
- **Default (W1-W2, W4-W6):** the snake is one rigid body; grounded iff it transitively
  rests on a `supports` cell. Falls uniformly toward -y; dead when *every* segment is below
  `floorY` (partial-void survives — oracle T-VOID-PARTIAL).
- **Anchor (W3+):** an anchored-on-grip segment is a grounding source, exactly like a
  world `supports` cell — it pins the body. No bridges, no per-segment weight, no chain
  physics. The anchor is an explicit grip on a *world* surface, and it grounds **only while
  the segment is on the grip cell**.
- **Multi-body (W7):** the same single fixpoint runs over `[snake, ...(bodies ?? [])]`. A
  body may rest on another body, but the chain must terminate at a real grounding source —
  mutual ungrounded pairs fall. The per-step early-win does **not** fire while `bodies` is
  present; `checkEnd` decides the all-heads win, **WIN before death** (D28).

**Mechanism-derived solidity never grounds (M1).** No entity whose `solid` is computed in
`applyMechanisms` (e.g. a `gate`) is ever a grounding source — by the PRESETS construction
guard such entities never set `supports` (§2.1, §2.4). Hence a mechanism change can never
alter the grounded set, and the single-pass `settle -> applyMechanisms` ordering is
correct.

**Termination (P-SETTLE-TERMINATION still holds):** each ungrounded component strictly
descends by 1 per iteration until it is grounded or has crossed `floorY`, so the fixpoint is
reached within `(maxY - floorY + 1)` iterations. A hard iteration cap inside `settle` asserts
the bound is never exceeded (§4.0, §7-note in §4.0).

**Oracles (added):** A-on-B-on-ground holds; A-on-B-on-A (mutual, ungrounded) both fall;
the settle result is **identical under any body ordering** (§4.4); a directed multi-body
termination/convergence oracle asserts a stacked/mutual config settles to the documented
fixpoint within the bound, checked on the **returned state** (T-SETTLE-CONVERGE, §4.4); a
`won` state passed to settle returns unchanged (T-WIN-FALL / monotonicity); and the co-op
win-wins-over-death precedence oracle (T-COOP-WIN-PRECEDENCE, §4.4). The
mutual-support-falls oracle is carried on the `bodies[]` ledger row now (§2.3).

### 2.6 World / level curriculum

Eight worlds, each: **introduce** (clean teach) -> **twist** (break the assumption) ->
**recombine** (finale that mixes the world's ability with prior ones). Target ~6
rooms/world, **~48 rooms** as a *ceiling, not a quota* — enough to teach the toolbox
without a content death-march. Per-room author cost is the real content risk (§5). A world
never introduces two new mechanics. Every room is a readable vertical plane.

**Authoring ergonomics land in Inc 1, not later.** Rooms are **drawn, not coordinate-typed**:

- **ASCII-art room format.** A room is rows of characters parsed into cells by a **pure,
  unit-tested parser** (e.g. `#`=wall, `o`=fruit, `X`=exit, `S`=snake, `.`=air; later
  worlds add glyphs for `grip`/`plate`/`gate`/`pickup`/`egg`). The parser produces a
  `LevelDef`; it is a pure function with a unit-tested oracle (§4.1, T-ASCII). **This is
  introduced before room ~5** (not "after ~40 rooms" — see D-OUT-2). The parser emits
  **preset names + overrides, never raw entity literals** (F6).
- **Per-room LEGEND BLOCK for id-bearing cells (F7, load-bearing).** A bare glyph map can
  draw stateless presets (`wall`/`fruit`/`exit`) but **cannot express id-bearing cells** —
  a `plate` with `trigger:'g1'` and the `gate` with `door:'g1'` it opens, or any later
  id-bearing entity. So the ASCII grammar includes an **optional per-room legend block**
  mapping a **glyph -> preset + overrides** (e.g. `a = plate trigger:g1`,
  `A = gate door:g1`; `b = plate trigger:g2`, `B = gate door:g2`). This makes
  **multi-pair Pressure rooms authorable** (distinct trigger/door ids per pair) and keeps
  the "rooms are drawn" claim honest for *all* worlds, not just the stateless ones. The
  legend feeds the same pure parser; T-ASCII gains a two-pair Pressure case asserting the
  grid parses to the correct **distinct** trigger/door ids (§4.1).
- **Record-par-from-INPUT-LOG.** A one-key shell action records the current play session as
  the room's par solution. **It serializes the INPUT LOG (the `(verb,dir)` move sequence),
  NOT the snapshot stack** (F3 — the snapshot stack is the *states*, not the *moves*). The
  Inc-1 shell maintains a parallel `inputs[]` array alongside the undo snapshot stack:
  every effective `act()` pushes `{verb, dir}`, and undo pops it in lockstep. The recorded
  serialized `(verb,dir)[]` feeds T-PAR and T-ROOM-SOLVE, which **replay that move
  sequence** (§4.5, §4.1). The author solves the room once, presses the key, and the
  recorded optimal move sequence is the par artifact.

### 2.7 Scoring model (eggs + move-par) — optional, non-blocking, outside the core

Eggs/par/constraints live **outside the pure core**, as functions over the snapshot trace
and the input log the shell already keeps — zero new core state, zero undo risk. The core
answers "is this move legal and what's the new world"; nothing about score.

- **Eggs (stars), max 3/room, all optional:** (1) **solve** — `status==="won"`;
  (2) **hidden** — touch a marked `egg` entity en route (collected like fruit but tracked,
  not grown; the flag is derived from the trace); (3) **constraint** — solve under a
  per-room predicate over the trace (`maxLength`, `maxMoves`, "no strike"). Eggs **loosely
  gate**: a world unlocks at a threshold the *solve* eggs alone exceed, so a player who
  only solves rooms always progresses (eggs gate *bonus* content, never the main path). A
  skip-after-K-failed-attempts path guarantees nobody is hard-walled.
- **Move-par:** the **hand-authored** optimal move count per room, stored in
  `LevelDef.par`, **shown only after first solve**. No solver is built (a solver is a
  project unto itself; cut). Par correctness is verified by an authoring oracle (T-PAR): a
  recorded optimal solution — the **serialized input log `(verb,dir)[]`** captured by the
  record-par tool (§2.6), **not the snapshot stack** — replayed by the test harness reaches
  `won` in exactly `par` moves, over *every* shipped room. **Caveat (honesty note):** T-PAR
  proves par is **achievable**, not provably **optimal** — par optimality is
  **author-asserted** (no solver verifies minimality). See §4.5, §6 D15.
- **Persistence:** one `localStorage` key (`coil.progress.v1`) holding highest world +
  egg set. Read/written through a tiny **injected store** so the serialize/parse
  round-trip is unit-tested off the real `window` (graft; §4.5). Move counter + egg
  progress live in `main.ts` (derived from `history.length` / the input log), **not in
  `GameState`** — so undo naturally rewinds them and they never pollute the pure core.

### 2.8 Renderer / animation approach

- **Clear-and-rebuild is the firm, permanent renderer decision.** Rooms are tiny; the
  current `render(state)` (dispose + rebuild meshes) is correct, simple, and already works.
  It is the **only firm renderer decision and the permanent fallback.** **No animation in
  Inc 1** — discrete snap-to-grid reads fine for a turn-based puzzle, and the slice's job is
  to prove the *feel of the rules*. Inc 1 adds only **camera auto-fit** (compute scene
  bounds from the room's authored bounds, set `controls.target`/distance) so rooms beyond
  level1's range frame correctly (resolves the hardcoded `(6,2,0)`).
- **Camera auto-fit is computed ONCE on room load (F-camera).** Auto-fit reads the
  **room's authored bounds** (the static `cells` extent + the snake's start), **not the
  live snake position**, and is computed **once at room load** — so framing is **stable**
  for the whole room and the **strike-into-a-fall risk stays visible** (the camera does not
  chase the snake out of frame at the decision moment). A snake that falls out of the frame
  is **acceptable** for these tiny rooms: the fall is the consequence the player committed
  to, and the room reloads/undo restores framing.
- **Animation is not committed.** *If/when* animation is later added (the first increment
  where multi-cell strike + mechanism reads are genuinely hard to follow as instantaneous
  redraws — practically Inc 2/3), it would use **persistent meshes** keyed by identity and
  **carve the snapshot-diff into a pure, tested function** at that point — but no
  `diffPlan.ts` file and no `T-DIFFPLAN` oracle are committed up front (that was
  over-engineering for a deferred, maybe-never feature). The renderer can **always fall back
  to a full clear-and-rebuild** on any diff it cannot plan — the permanent safety valve.
- **Strike trace (render-only, if animation lands):** `move`/`strike` could optionally
  return `{state, trace}` where `trace` is the ordered intermediate head cells,
  **render-only metadata that never enters the snapshot or the undo stack** (graft from
  draft-risk §2.4). `main.ts` would feed `trace` to the renderer and store only `state` in
  history; undo never sees a trace. This is a fallback if curved/interrupted strikes ever
  make renderer-side path reconstruction ambiguous — not built for Inc 1.
- **Any future animation work is purely additive to the renderer** — it touches no other
  layer and does not gate Inc 1. If it gets hairy, clear-and-rebuild is a permanently
  acceptable fallback.
- **Entity rendering generalizes trivially:** the renderer switches on `entity.kind` for
  mesh/color and reads `heat` for the Dark lighting mode. No rules leak in.
- **Snake look:** Inc 1 keeps boxes (head distinct). A tube/oriented-segment look is a
  nice-to-have slotted with any future animation work, not a blocker.

## 4. Acceptance criteria / test oracles (definition of done)

The test plan **drives the model**: a feature is not "done" until its oracle is written
(red) then green. All core oracles are **vitest** (node env), pure `state -> assert`,
deterministic (no clock; randomness only via fast-check from Inc 3, §4.0), run with
stdin closed. Renderer/almanac oracles are partly **scripted** (the one hard ship gate,
§2.9) and partly a **manual checklist** for the irreducibly visual checks.
**Definition of done for an increment = all its oracles pass + `tsc --noEmit` clean.**

### 4.0 Core-wide property battery — from Inc 1, forever

These invariant oracles are re-run against **every verb** — the cheapest, highest-leverage
testability multiplier.

**Inc 1: a COMMITTED FIXED ADVERSARIAL CORPUS, not fast-check (F5/M4).** At Inc 1 the
battery is driven over a **committed `CORPUS: GameState[]`** — the documented edge cases
(blocked-by-wall, blocked-by-body, tail-vacate, head-on-exit, no-support, snake half over
void, high-snake-over-deep-void, multi-segment fall, etc.) **plus pinned regression seeds**
(any state that ever found a bug is kept forever). **A *defined adversarial* corpus answers
round-1 F15's vacuity concern WITHOUT a new dependency** — example-based properties over a
deliberately adversarial committed corpus are not "examples wearing a property hat" when
the corpus is chosen to attack the invariants. **This REFINES F15 (defined corpus now,
randomization later); it does not reverse it.** **Verb-registration stays the gate:** every
verb (including future verbs) MUST be **registered** with the battery; the registration
list is what the P-* properties iterate over the corpus. Adding a verb without registering
it is a visible omission.

**T-PRESETS (Inc 1, build-time assert — F6/M1).** A test over the frozen `PRESETS` table
asserts: **(a)** no preset sets a **forbidden flag combination** (e.g. `win+solid+eat`, or
any combo the design declares nonsensical); **(b)** the **M1 invariant** — every preset
whose solidity is **mechanism-derived** (e.g. `gate`, recomputed in `applyMechanisms`) has
**`supports` falsy**. Combined with the structural rule that `LevelDef` and the ASCII parser
reference presets **by name only** (no raw entity literals in level data — §2.1/§2.6),
"construct only via PRESETS" is structurally near-enforced rather than mere convention.

**fast-check is introduced ONLY where input-space exploration genuinely pays:**
- **Inc 3** — the T-RESOLVE-EQUIV differential corpus (frozen OLD tail/fixtures vs NEW
  `resolve`), where a randomized state space is the whole point of a differential oracle
  (§4.3, D21). `fast-check` is added as a **TEST-ONLY devDependency here** (§3 package.json
  row).
- **Inc 4** — carry / co-op state generation (the input space genuinely widens).

The Inc-3 generator is a seeded, deterministic `fast-check` `Arbitrary<GameState>` over
small grids (random snake placement, random preset cells, random `strikeRange`/`floorY`),
with **pinned regression seeds**, and it **grows** to cover `carry` / active `body` when
those land. Settle property-fuzzing **may be re-enabled once `bodies[]` lands** (Inc 4),
when settle becomes non-trivial (M6).

Properties (driven over the Inc-1 corpus, then additionally over fast-check from Inc 3):

- **P-DETERMINISM:** `v(s, dir)` is byte-identical (deep-equal) on repeated calls, over the
  corpus.
- **P-PURITY:** after `v(s, dir)`, `s` deep-equals a pre-call clone (no mutation).
- **P-NOOP-IDENTITY:** a blocked/terminal action returns the **same reference** (`toBe`),
  so the shell's `next === state` no-op detection holds.
- **P-UNDO:** apply `v` then restore the previous snapshot deep-equals the original, for
  every registered verb (the snapshot stack is the undo spec). Extended to non-null `carry`
  and to an `active`-body switch as those land.
- **P-CONSERVATION (scoped to what exists at each increment):** at **Inc 1**, every
  non-eating action leaves `snake` length unchanged and an eating action is exactly +1.
  **Extended at Inc 4** to carry/shed per the ledger (a shed conserves body + deposited
  matter; carry adds 0 to length).

**Settle termination is a DIRECTED test, not a fuzz property (scope decision, see C7/D-note):**

- **P-SETTLE-TERMINATION (directed):** a high snake over a deep void reaches `dead` in the
  expected iteration count (an exact directed assertion), **PLUS** a **hard iteration-cap
  assertion inside `settle`** that fires if the `(maxY - floorY + 1)` bound is ever exceeded.
  Property-fuzzing of settle is **deferred** until a mechanic makes settle non-trivial
  (re-enabled once `bodies[]` lands, M6). The Inc-1 corpus / Inc-3 generator serve the other
  P-* properties.

  the corpus; **OR** equivalently `OLD settle(s)` vs `NEW checkEnd(settle(s))`. **The
  mismatched `settle(checkWin(s))` comparand is dropped** (the C3 win-check move broke the
  old pairing). This proves the `resolve` tail extraction is behavior-preserving against a
  comparand that genuinely predates the change. (All Inc 0-2 directed tests also pass
  unchanged.)
- **T-MECH-1** *(RELEASE case, gate cell EMPTY)* a segment on a `plate` opens its `gate`
  (`solid` false) after settle; stepping off **with the gate mouth empty re-closes it**
  (derived from triggers + occupancy each turn, NO latched state).
- **T-MECH-2** *(OCCUPIED case)* with the gate mouth **occupied by a segment**, the gate
  **stays open and there is no crush** (occupancy holds it open; no cross-turn latch).
- **T-MECH-3** mechanism resolution is deterministic and undoable (replay -> identical);
  occupancy is an existential test over ANY segment of ANY body, so it is order-independent
  across bodies.
- **T-BUILD (mechanism case, F10)** a level authored with a segment starting on a plate
  **builds gate-open** (`buildState` runs the full `resolve` tail from Inc 3).
- **Dark (M-DARK, manual):** in dark mode only heat sources + snake + a small radius
  render; rules behave identically to lit (a known room solves identically). **Core
  regression:** all prior tests pass byte-identically with heat cells added.

### 4.4 Increment 4 — carry (W6) / decoy / co-op (W7)
- **T-CARRY-1** stepping onto a `pickup` stores it as `carry` (length unchanged, cell
  cleared); swallow with full gut -> no-op (same ref).
- **T-CARRY-2** depositing places the entity in the target empty cell as a normal entity;
  a deposited `solid` then blocks/supports; deposit into a blocked cell -> no-op (same ref).
- **T-DECOY** a shed block acts as static structure (supports / holds a plate);
  matter-conservation holds.
- **T-SETTLE-MUTUAL** *(transitive grounding oracles)* A-on-B-on-ground **holds**;
  A-on-B-on-A (mutual, ungrounded) **both fall**; the settle result is **identical under
  any body ordering** (order-independence by construction).
- **T-SETTLE-CONVERGE (M6, directed multi-body termination/convergence)** a stacked / mutual
  multi-body configuration settles to the **documented fixpoint within the
  `(maxY - floorY + 1)` bound**, asserted on the **RETURNED state** (the exact resting
  positions), **not merely "did not throw."** *(Settle property-fuzzing may be re-enabled
  here now that `bodies[]` makes settle non-trivial.)*
- **T-COOP-WIN-PRECEDENCE (D28, win-wins-over-death)** a multi-body fixpoint in which **one
  body's head reaches its exit while another body would fall into the void on the SAME
  turn** resolves to **`status === 'won'`**, NOT `'dead'` — within `checkEnd`, WIN is
  evaluated before death across bodies.
- **T-COOP-MIGRATE** *(the migration gate, M2)* an Inc-0 single-snake golden trace replayed
  through the multi-body kernel (with `bodies` **absent / `undefined`**, never written back)
  yields **byte-identical** snapshots.
- **T-COOP-1** *(if W7 shipped)* switching bodies moves the active pointer; verbs affect
  only the active body; one body supports the other (transitive grounding), each settles
  in the single simultaneous fixpoint.
- **T-COOP-2** *(if W7 shipped)* win requires **all** heads on exits **(decided by
  `checkEnd`; the per-step early-win does not fire while `bodies` is present — D28; WIN
  before death — T-COOP-WIN-PRECEDENCE)**; any body fully in void -> dead **only if no
  all-heads win occurred this fixpoint**; settle/mechanisms run over all bodies.
- **Spikes I4a (carry) and I4b (co-op) green before authoring W6/W7.**

### 4.5 Scoring oracles
- **T-PAR** for *every* room, a recorded optimal solution — the **serialized input log
  `(verb,dir)[]`** captured by the record-par tool (§2.6), **NOT the snapshot stack** —
  replayed reaches `won` in exactly `LevelDef.par` moves (catches a wrong/now-impossible par
  after any engine change). **Caveat:** this proves par is **achievable**, not provably
  **optimal** — par optimality is **author-asserted** (no solver is built).
- **T-EGG-HIDDEN** a scripted solution touching the marked `egg` flags it; one avoiding it
  does not.
- **T-EGG-CONSTRAINT** the constraint predicate is true for a compliant solve, false for a
  violating one.
- **T-GATING** the world-unlock threshold is reachable by *solve eggs alone* (no main-path
  wall); the skip-after-K path also unlocks.
- **T-PERSIST** the `coil.progress.v1` serialize/parse round-trip is unit-tested through
  the injected store (off the real `window`).

**Par/solve stability policy (load-bearing).** A recorded solution (par or solve) going
**RED under an engine change is BY DEFAULT a REGRESSION to investigate, NOT a signal to
re-record.** Re-recording a par/solve artifact is allowed **only** when the change is an
**intended rule change carrying its own oracle** (the new behaviour is itself pinned by a
test). This keeps T-PAR / T-ROOM-SOLVE **guards**, not rubber stamps — a red is treated as
"the engine broke a room" until proven to be "the rule deliberately changed."

### 4.6 Renderer oracles
- **M-RENDER (manual)** the renderer constructs without throwing for a tiny state; the

  on load and does not chase the snake (F-camera, §2.8). *(Irreducibly visual — stays
  manual.)*
- *(No committed `diffPlan` / `T-DIFFPLAN`: if/when animation is added, carve the
  snapshot-diff into a pure tested function at that time — §2.8.)*

## 5. Residual risks

1. **Entity refactor regresses the slice.** Mitigated by **ordering**: the §4.0 property
   battery (driven over the committed fixed adversarial corpus) + §4.1 oracle-gap tests are
   landed GREEN on the **pre-refactor** code FIRST, then the refactor is done against that
   hardened net (the refactor is **not** the first Inc-1 commit — the hardening is). The
   refactor is mechanical (presets reproduce the three types exactly) and the existing + new
   tests are the oracle; because they were proven green on the old code, any post-refactor
   red is unambiguously the refactor's fault. **The Inc-1 split option (§1.3) further
   de-risks this** — the refactor can ship first, before the authoring pipeline. T-RESOLVE-
   EQUIV (Inc 3, against the frozen pre-extraction comparand — D21) adds a differential
   check for the tail extraction.
2. **Animation scope creep — pre-empted.** No animation and no `diffPlan` module are
   committed; clear-and-rebuild is the firm, permanent decision and fallback. If animation is
   ever added it is renderer-isolated and carves out a pure tested diff function at that time
   (§2.8). The risk is deferred to a maybe-never feature, not carried now.
3. **Co-op (W7) cost.** Even minimal, multi-body touches the state root, settle/mechanism
   passes, and input. Mitigated: it is **cut-first**; done **last**; the settle is the same
   transitive-grounding fixpoint (order-independent, mutual-support-falls, win-wins-over-
   death, T-SETTLE-CONVERGE), `bodies` stays absent for single-snake (M2), and it is guarded
   by the golden-trace migration oracle (T-COOP-MIGRATE) and a spike. The game is complete
   without it. **The fixpoint/body-graph machinery is built HERE, not at Inc 1 (D30)** — no
   on-ramp to nowhere.
4. **Move-par authoring burden.** Hand-authored par needs a recorded optimal per room —
   captured cheaply by the record-par tool, which serializes the **input log** `(verb,dir)[]`
   (not the snapshot stack — §2.6). Mitigated by T-PAR (a wrong par fails the build) and
   showing par only post-solve. T-PAR proves *achievable*, not *optimal* (author-asserted).
   A par/solve going red is by default a regression to investigate, not a re-record signal
   (§4.5). No solver is built (YAGNI at ~48 rooms).
5. **(Formerly: constrict may be a feature nobody needs.) RESOLVED — formal constriction is
   CUT (D29).** No shipped world needs formal crushing; the `surrounds` predicate and
   `crushable` entity are removed, not deferred. Wrap stays in the toolbox as emergent
   body-blocking (no model addition). If a future world genuinely needs crushing, the
   predicate is defined then, against that concrete room (§2.2.5). This removes both the
   speculative feature and its unbounded-map border-definition problem.
6. **Content production is the real long pole**, not engineering — and it is now an
   **explicit go/no-go gate.** ~48 hand-designed, solvable, fun rooms is the bulk of the
   work; the model is small by design precisely so effort goes here. **The Inc-1 playtest
   gate (D24, §1.3) is the kill-criterion, now a concrete pass/fail bar:** 3–5 external
   players, each must learn move/strike/grow/gravity unaided and reach `won` on every W1
   room without a hint; GO requires >=75% (all-but-one) pass with no room abandoned in
   frustration; Arthur (or a named facilitator) runs it. If it fails, the project stops at
   the complete W1 game and the `[DRAFT]->[PUB]` flip is a real decision. **Content
   ownership is an explicit OPEN success-criterion** (§2.6): *who* authors the rooms and
   whether one person can realistically do it. Mitigated by the ASCII room format + legend
   block + record-par tool landing in Inc 1 (optionally split to Inc 1b, §1.3), per-world
   increments (value lands continuously), spike-before-content for structural worlds, and
   T-PAR catching broken authoring. **"Fun"/"difficulty" are validated by the human PLAYTEST
   gate, not an oracle**; T-ROOM-SOLVE ("solvable") is the machine-checkable floor.
7. **(Formerly: undo depth — open question #7.) RESOLVED.** The undo snapshot stack and the
   parallel `inputs[]` are **UNBOUNDED within a room** and **reset on restart / room-nav**
   (§2.7) — rooms are small, so no cap is needed. This explicitly closes the brief's
   open-question #7.
8. **Bundle size (`three` + OrbitControls).** Unmeasured. **No build-breaking budget at
   Inc 1 (M5):** the gzipped bundle size is **measured and recorded in `coil/SOURCE.txt`**
   as informational data; a size gate (warning, or an absolute ceiling + margin) is added
   **only if a regression actually appears.** Same risk night-cab already accepted; import
   only used `three` modules, match night-cab's `three` version; not expected to block a
   Pages doc. The no-absolute-`/assets/` grep is the one hard scripted ship gate.
9. **Anchor as a third control.** Slightly grows the surface. Mitigated: it's a *toggle*,
   introduced only in W3 with a dedicated teach room; two movement verbs + one toggle still
   fits in the head. The anchor toggle and the carry deposit modifier share a key on the
   no-coexist assumption; a room must EITHER not combine them OR the build assigns distinct
   keys (F8, D8).
10. **Darkness-as-render could disappoint** if players expect fog to be a "rule."
    Mitigated: it's a deliberate testability trade; the puzzle (limited-info navigation) is
    unaffected; reversible later (it would become a real model change with its own oracle).
11. **Two-repo mirror drift.** `coil/` in the almanac is build-output-only and never edited
    there; the recorded source SHA (`coil/SOURCE.txt`) makes any almanac<->mdsnake HEAD
    divergence detectable. A single-repo alternative was considered and rejected (D3); the
    mirror discipline + SHA check is the mitigation.
12. **New test dependency (`fast-check`), deferred to Inc 3.** Inc 1 needs **no new
    dependency** — its property battery runs over a committed fixed adversarial corpus
    (F5/M4). `fast-check` is a proposed TEST-ONLY devDependency introduced **at Inc 3** for
    the T-RESOLVE-EQUIV differential oracle (and extended to carry/co-op at Inc 4), where
    input-space exploration genuinely pays. Mitigated: test-only (never ships in the bundle),
    and it is the one place randomness enters the otherwise-deterministic test suite.
