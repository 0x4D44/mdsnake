# HLD — Coil: a snake puzzle game

Status: DRAFT V2
Date: 2026-06-20

**Angle:** ruthlessly minimal core, hardened with mechanically-checkable oracles.
One unifying primitive (an entity of orthogonal flags) collapses most abilities into
*data*, not new code paths. Every concept must earn its place; defer or cut anything the
next shipped world does not demand. A property/invariant test battery and a "no on-ramps"
ledger make the discipline reviewer-checkable; structural extensions are de-risked by a
throwaway spike before any world rides on them.

**Authoritative source repo:** `C:\language\mdsnake` (the night-cab analogue), mirrored
into the almanac as `coil/`. **`coil/` in the almanac is BUILD OUTPUT ONLY and is NEVER
edited there;** any hot-fix goes to `mdsnake` first, then rebuild + recopy (mirror
discipline recorded in the almanac `CLAUDE.md`; see §2.9 and D3).

This HLD synthesizes three competing drafts. It takes the simplest-viable model as the
spine and grafts in the strongest verification artifacts and de-risking disciplines from
the runners-up. The Decision Log (§6) records every load-bearing choice.

---

## 0. Thesis (the one idea to hold in your head)

The scaffold already has the right spine: a **pure, immutable, discrete sim core**
(`state -> state`), a **single rigid snake under gravity**, and **two verbs**
(`move`, `strike`). Almost the entire ability toolbox is expressible by changing *one*
thing: the closed `CellType` union becomes a small **tagged entity with orthogonal
property flags**. Walls, fruit, exits, and most future abilities (anchors, heat, doors,
carried items, decoys) are then *data*, not new code paths.

The honest framing of that win (see §2.3, D6): the payoff is that the **existing rule
call-sites (step / support / eat / win) stop needing per-ability branches** because they
read flags. A *new* entity is not free — it costs one `PRESETS` row + one renderer `kind`
branch + (for active behaviours) one resolution pass or verb. What the flag-set buys is
that the hot rule functions never grow a `switch` per ability; that is the simplicity win,
not "abilities become free."

Two abilities genuinely demand a new structural concept and nothing else does:
**per-segment grip** (anchoring, World 3) and **a second snake** (co-op, last and
cut-first). Each gets a spike before content.

That is the whole design. The rest is content and verification. The single rule that
makes it all work — and that every ability preserves — is:

> **The rules are pure functions `state -> state`; the test suite IS the spec.**
> Determinism + immutability give undo-as-snapshot-stack, referential-equality no-op
> detection, and unit tests that read like a truth table. Every ability is admitted to
> the model **only** with an oracle that pins its behavior first.

## 1. Scope

### 1.1 In scope (this HLD designs the whole game)

- A **core data-model evolution** expressing the full toolbox with the fewest concepts,
  each extension pinned to the world that first needs it (and *not before*), recorded in a
  reviewer-checkable "no on-ramps" ledger (§2.3).
- The **complete ability toolbox**: body-as-structure, strike/coil, length/growth,
  wrap/constrict, rattle (redesigned, agent-free), heat-sense/dark, swallow & carry,
  shed-skin decoy, co-op/relay.
- The **world curriculum** (8 worlds), sequenced into independently shippable increments.
- The **scoring model**: eggs (solve / hidden / constraint) + move-par, kept optional,
  non-blocking, and entirely outside the pure core.
- The **renderer/animation** plan: clear-and-rebuild is the firm, permanent decision; the
  single isolated place a persistent-mesh pivot *might* later be justified is a one-line
  note, not a committed module.
- The **almanac ship plan** (night-cab style, Vite `base:"./"`).
- The **acceptance oracles** (property battery + per-mechanic directed tests) that define
  "done" for each increment.
- **Authoring ergonomics from Inc 1**: a tiny ASCII-art room format with a pure, tested
  parser, and a "record the current play session as par" tool in the shell (§2.6, §6 D23).

### 1.2 Out of scope (explicitly cut or deferred — see Decision Log)

- **Touch / gamepad / on-screen controls / accessibility / key remap.** Desktop keyboard
  + mouse-orbit only, as today. (D-OUT-1)
- **JSON level format + in-browser editor.** Levels stay TS literals (authored via the
  ASCII room format, §2.6). `LevelDef` is already plain serializable data, so promotion to
  JSON is a cheap mechanical change later — do it *only if* authored room count genuinely
  outgrows the TS-literal-plus-ASCII path. (D-OUT-2)
- **Coil-to-charge** (strike range scaling with length). **Cut.** `strikeRange` stays a
  level scalar. Length already gives reach as *structure* (a longer body spans more
  cells); variable reach adds a feel parameter and tuning burden no shipped puzzle needs.
  Re-evaluate only if a world's puzzle genuinely requires variable reach. (D-OUT-3)
- **A creature AI / pathfinding / sound-propagation system.** The "Rattle" world is
  redesigned around a deterministic, snake-triggered plate->gate hazard (no autonomous
  agent). (D-OUT-4)
- **Full multi-snake co-op as a separate game mode.** Folded into a minimal "second body"
  model and flagged as the first thing to cut if production runs long. (D-OUT-5)
- **3D/depth puzzles.** `Vec` stays `{x,y}`; the third axis is presentation-only forever.
- **Server-side anything, audio engine, timers, lives, accounts, leaderboards,
  telemetry.** "Rattle" ships silent with a visual pulse; real sound FX are a later
  cosmetic pass at most.
- **Progress persistence beyond one `localStorage` key** (`coil.progress.v1`: highest
  world unlocked + per-room egg flags). No cloud sync, no migration (sandbox).

### 1.3 Delivery sequencing (increments)

Each increment is independently shippable: it compiles (`tsc --noEmit` clean), passes
`vitest`, builds to a working `dist/`, and is publishable to the almanac under the
subpath. **Increment 1 is already built and tested** (the slice) and ships with zero new
model surface beyond the hardening below.

| Inc | Ships | New model surface | Worlds |
|-----|-------|-------------------|--------|
| **0** | *(done)* Slice: move + strike + grow + gravity + exit/void, Room 1, full vitest. | none | W1 R1 |
| **1** | **First public ship.** **First** land the §4.0 property battery + the §4.1 oracle-gap tests GREEN on the pre-refactor code; **then** do the entity refactor (behaviour-preserving) against that hardened net; add the ASCII room parser + record-par tool; build World 1 (Hatchling, 5-7 rooms); undo-depth + restart polish; camera auto-fit; ship to almanac as `[DRAFT]`. | none new (entity-flag refactor of the existing three types) | W1 |
| **2** | World 2 (Growth) + World 3 (The Climb). | `grip` flag + per-segment `anchored`; the `anchor` toggle verb. **Spike first.** | W2, W3 |
| **3** | World 4 (Pressure) + World 5 (Dark). | `trigger`/`door`/`heat` flags; the post-settle `applyMechanisms` pass; the shared `resolve` tail extracted here. Dark is renderer-only. | W4, W5 |
| **4** | World 6 (The Gullet) + World 7 (Two Bodies, *minimal, cut-first*) + Finale. | `pickup`/`egg`/`carry` on a segment; optional `bodies: Segment[][]`. **Spike carry and co-op first.** | W6, W7, W8 |

> **Inc-1 ordering (load-bearing).** The hardening net is landed **before** the refactor,
> not after: harden first against the known-good pre-refactor code, *then* refactor against
> a net that already passes. This breaks the circular "the tests pass, therefore the
> behaviour is preserved" reasoning — the net is proven green on the old code first, so a
> later red is unambiguously the refactor's fault. See §2.1 and Residual-risk #1.

If only Increments 0-1 ever ship, Coil is a complete, satisfying small game. Everything
after is additive content on the same engine. **The render/animation pivot (§2.8) is
deferred to Inc 2+ and is renderer-isolated** — it never gates the first public release.

## 2. The design

### 2.1 Core data model — the one unifying change

**Today** (`types.ts`):

```ts
export type CellType = "wall" | "fruit" | "exit";   // closed union
cells: Map<string, CellType>;                        // one tag per cell
```

The closed union forces every new ability into either a new string variant *and* matching
branches in `tryStep`/`settle`/`checkWin`, or a parallel structure. That is the "many
special cases" trap.

**The change** — replace the string tag with a tiny **entity record of orthogonal
properties**. The properties are exactly the questions the rules already ask:

```ts
export interface Entity {
  kind: EntityKind;     // for the renderer + readability; rules read flags, not kind
  solid?: boolean;      // blocks a step (today: wall)
  supports?: boolean;   // bears the snake's weight from below (today: wall)
  eat?: boolean;        // stepping onto it grows the snake & removes it (today: fruit)
  win?: boolean;        // head on it -> won (today: exit)
  // --- added only when its world arrives (the §2.3 ledger says when) ---
  grip?: boolean;       // snake may anchor a segment here          (Inc 2 / World 3)
  heat?: boolean;       // renderer-only: visible in the dark        (Inc 3 / World 5)
  trigger?: string;     // a plate; pressing it fires a mechanism id (Inc 3 / World 4)
  door?: string;        // a solid this trigger id toggles open      (Inc 3 / World 4)
  pickup?: boolean;     // swallowable; stored on a segment as carry (Inc 4 / World 6)
  egg?: boolean;        // a collectible scoring marker (no growth)  (Inc 4 / scoring)
}
export type EntityKind =
  | "wall" | "fruit" | "exit"
  | "anchor" | "heatlamp" | "plate" | "gate" | "object" | "egg";  // grows with increments

cells: Map<string, Entity>;
```

**Why a flag-set and not the closed union (the contested call, resolved):** one judge
preferred keeping a closed `CellType` union with a `cellProps()` lookup, to forbid
nonsensical cells (`win+solid+eat`). We adopt the flag-set as the spine because it is the
single primitive that keeps the **hot rule call-sites branch-free** (they read flags, not
a per-ability `switch`), and we **graft the invalid-combination guard**: a small frozen
`PRESETS` table is the *only* sanctioned way to construct an entity, so authors pick a
named preset, never raw flags. This gives the flag model's extensibility without flag-soup
invalidity. (Honest framing: a *new* entity still costs a `PRESETS` row + a renderer `kind`
branch + any active resolution pass — see §2.3 and D6/D10.)

```ts
const PRESETS = {
  wall:  { kind: "wall",  solid: true, supports: true },
  fruit: { kind: "fruit", eat: true },
  exit:  { kind: "exit",  win: true },
  // added by the increment that needs them:
  anchor: { kind: "anchor", solid: true, supports: true, grip: true },   // Inc 2
  plate:  { kind: "plate", trigger: /*id set per cell*/ "" },            // Inc 3
  gate:   { kind: "gate",  solid: true, door: "" },                      // Inc 3
  // ...
} as const;
```

- The three existing types collapse to presets *with no rules change to their meaning*.
- The rules read **flags, never the kind string**: `tryStep` blocks on `e.solid`;
  `isSupported` checks `e.supports`; eating checks `e.eat`; `checkWin` checks `e.win`.
  This is a *mechanical refactor* of the existing four call-sites — same logic, one
  indirection.
- **Inc-1 ordering of the refactor (load-bearing).** The §4.0 property battery and the
  §4.1 oracle-gap tests are landed GREEN on the **pre-refactor** code first; the entity
  refactor is then done against that hardened net. The *existing + new* tests assert
  behavior, not representation, so they must stay green across the refactor — and because
  the net was proven green on the old code, a post-refactor failure is unambiguously the
  refactor's fault. This is the Inc-1 acceptance gate. For the pure-tail equivalence
  specifically, T-RESOLVE-EQUIV adds a **property-based differential oracle** (§4.3): a
  random-state corpus is run through the OLD tail `settle(checkWin(s))` and the NEW
  `resolve(s)` (mechanisms absent) and asserted deep-equal.
- **No flag is added until the increment that uses it.** `grip`/`heat`/`trigger`/`door`/
  `pickup`/`egg` are listed above for foresight but are absent from the Inc-1 type and
  presets. The foresight lives in the *naming convention* (orthogonal booleans), not in
  unused code — see the ledger (§2.3).

This single change carries anchors, heat lamps, pressure plates + gates, swallowable
items, and decoys *without growing the hot rule functions*: the existing call-sites read
flags and never branch per ability. New entities still cost their PRESETS row + renderer
branch + any active pass (§2.3).

#### State shape (final, Inc-4)

```ts
export interface GameState {
  snake: Segment[];     // ordered, head first  (was Vec[])
  cells: Map<string, Entity>;
  strikeRange: number;
  floorY: number;
  status: Status;       // "play" | "won" | "dead"
  name: string;
  // added only by the increment that needs it:
  triggers?: Set<string>;    // currently-pressed trigger ids (Inc 3)
  bodies?: Segment[][];      // additional snakes for co-op   (Inc 4, may be cut)
}

// Inc 1: Segment === Vec (a bare {x,y}); the alias keeps Inc-1 byte-identical.
// Inc 2 widens it:
export interface Segment { x: number; y: number; anchored?: boolean; carry?: Entity; }
```

> **Migration note (sandbox: no back-compat owed).** `snake: Vec[]` becomes
> `snake: Segment[]` with `Segment` a superset of `Vec`, so Inc-1 code is unchanged.
> `LevelDef.cells` becomes a list of `{x, y, preset, ...overrides}`; `buildState` builds
> the entity map from presets. The whole change is type-additive. We do **not** rename
> `snake` to a `snakes[]` array up front (a runner-up did, then listed it as its own
> residual risk — an on-ramp to nowhere); co-op widens the container last, behind a
> golden-trace equivalence oracle (§2.2.10, §4.9).

### 2.2 How every ability is expressed

The toolbox, each mapped to the **smallest** model addition. "New verb?" tracks the
control-surface budget: the design strongly prefers **zero new verbs** (only `move` and
`strike` exist) and expresses abilities as *board state the existing verbs interact with*.

| Ability | Model addition | New verb? | Increment |
|---|---|---|---|
| Body-as-structure | none (emergent from snake occupancy + rigid fall) | no | 0 |
| Strike / coil | none (`strike` verb already present) | no | 0 |
| Length / growth | none (`eat` flag already present) | no | 0 |
| Wrap / constrict | a pure `surrounds` predicate (geometry, not stored) | no | 2 (cut candidate) |
| Anchor (was "climb") | `Entity.grip` + `Segment.anchored` + one toggle | **1 verb: anchor** | 2 |
| Rattle (redesigned) | `Entity.trigger`/`door` + a mechanism pass | no | 3 |
| Heat-sense (dark) | `Entity.heat` (renderer-only) | no | 3 |
| Swallow & carry | `Segment.carry` + eat-of-`pickup` entity | no (deposit modifier) | 4 |
| Shed-skin decoy | deposit a body-shaped `solid` from carry | reuses carry | 4 |
| Co-op / relay | `bodies: Segment[][]` (minimal) | no (Tab cycles) | 4 (cut-first) |

The conscious control-surface budget for the entire game is **two movement verbs
(`move`, `strike`) + one toggle (`anchor`) + a deposit modifier + Tab (co-op only)**.

#### 2.2.1 Body-as-structure (W1, exists)
No model. The snake's own segments already block steps (`tryStep` body check) and occupy
space; a coiled body is a wall you made. Puzzles use this directly: park your tail to
stand on later, or to block a fall, or to bridge a gap with the C-shape (tail-vacate).

#### 2.2.2 Strike / coil (W1, exists)
No model. `strike(dir)` launches the head up to `strikeRange` with gravity suppressed,
then one settle. **Resolved open questions (now acceptance oracles, §4):**
- *Strike ending over void:* the final `settle()` runs normally — a strike that lands
  over a gap **falls and may die**. Intended (it's the risk that makes strike a
  decision). Oracle T-STRIKE-3.
- *Strike over fruit mid-flight:* the head **eats and grows mid-strike** (a `tryStep` onto
  `eat` keeps the tail). Intended; it lets a strike both cross a gap and grab a fruit.
  Oracle T-STRIKE-4.
- *Win mid-strike:* if the head crosses an exit cell during the strike flight, the game is
  won **at that instant** — `checkWin` runs on each strike flight step (§2.3, §2.4).
  Oracle T-WIN-STRIKE-CROSS (§4.1).

#### 2.2.3 Length / growth (W1-W2, exists)
No model. `eat` entities grow the snake (+1, tail kept). Growth's puzzle role: longer =
reaches farther *as a structure* and fills more cells (easier to trap yourself).
**Coil-to-charge is cut (D-OUT-3)** — length does not change `strikeRange`. Reach-from-
length is already expressible: a longer body physically spans more cells.

#### 2.2.4 Anchor (W3 "The Climb") — the first real structural addition
The only sub-rigid-body concept the game needs. An `anchor` entity carries `grip:true`. A
new verb:

```ts
anchor(s): GameState     // toggle: anchor/release the segment currently on a grip cell
```

When a segment is `anchored`, **`settle()` does not fall** (the snake hangs). This folds
into the transitive-grounding settle (§2.5): an `anchored` segment is a grounding source,
exactly like a world `supports` cell. Climbing is then `anchor -> move head up along a grip
wall -> re-anchor -> release tail`. This expresses **perching/gripping** without a per-cell
grip graph or rope physics.

We explicitly **reject per-segment partial support** (a snake half-hanging off a ledge
sagging under its own weight). That is the un-holdable full-3D knot the design forbids;
anchoring is binary (you grip or you don't), keeping `settle` a short function forever.

> **Why a verb here and nowhere else:** anchoring is a *choice the player makes about
> their own body* that gravity must read — it cannot be board data alone. Every other
> ability is board data the existing verbs touch, so they cost no verb. Two movement verbs
> plus one toggle is the whole control surface. Hold-in-head test: passes.

**Spike before content (R14 graft):** before authoring World 3, build one throwaway room
proving a 3-cell anchored snake holds while the head climbs/strikes. If the feel is
wrong, only the spike is lost, not a world.

#### 2.2.5 Wrap / constrict (W2 twist) — geometry, not a rule
No model addition. "Constrict" = surround a target entity so it counts as crushed/held.
This is **detected**, not stored: a pure predicate `surrounds(snake, cell): boolean`,
defined unambiguously as **air-flood-fill reachability to the grid border** — the target
is constricted iff a flood-fill from its cell across air cannot reach the border without
crossing a body segment (graft from draft-testability: removes the figure-eight /
loop-touching-wall ambiguity). A `crushable` entity (a `solid` that disappears when
surrounded) reuses the entity-flag system. **Cut candidate** — build only if a W2 twist
genuinely needs it; else delete the predicate.

#### 2.2.6 Rattle (W4 "Pressure"), redesigned to kill the AI dependency
The original "rattle = sound triggers a creature" needs a creature/AI/sound-propagation
model — three systems, none of which fit "hold it in your head," and the highest-scope
thing the runner-up drafts kept building. **Redesign:** keep the *theme* (your body's
motion triggers the environment) but make it **deterministic and agent-free**:
- A `plate` entity has `trigger:"g1"`; a `gate` entity has `door:"g1"` and `solid:true`.
- After settle, a **mechanism pass** computes
  `triggers = { ids of plates currently under a snake segment }`, then derives **every
  turn, from scratch**:

  ```
  gate.solid = !(gate.door in triggers) && !(gate cell occupied by any segment of any body)
  ```

  A gate is **held open by EITHER its plate OR a body in its mouth**; its solidity is
  *derived each turn from current triggers + occupancy, with NO cross-turn latched state*.
  Pure, deterministic, undoable. **Occupancy is an existential set-membership test**
  ("occupied by ANY segment of ANY body") and is therefore **order-independent across
  bodies**.
- "Rattle" flavor: stepping on a plate opens a distant gate; the puzzle is holding a plate
  down *with part of your body* while reaching the gate with the rest — body-as-structure
  again, with a visual pulse for the "rattle."

This is one pure function (`applyMechanisms(s): s`) in the shared `resolve` tail. No
autonomous entities ever. (D-OUT-4.) **Gate solidity is purely derived (no latch):** when
the plate releases AND the gate mouth is empty, the gate re-closes (oracle T-MECH-1); when
the gate mouth is occupied by any segment, the gate stays open and there is **no crush**
(oracle T-MECH-2). There is no triggers-only formula — occupancy is always part of the
derivation.

#### 2.2.7 Heat-sense / Dark (W5) — renderer-only, zero rules
`Entity.heat:true` marks lamps/warm cells. The **core does not change at all** — `heat` is
read solely by the renderer, which dims everything except heat sources, the snake, and a
small radius. All collision/gravity/win rules are identical (regression oracle: every
prior core test passes byte-identically with heat cells sprinkled in). The "dark" puzzle
is purely about *information*: remember geometry you saw lit. Making fog a rule would add
hidden state and wreck testability for zero puzzle benefit. The cheapest world: content +
a renderer lighting mode (still one renderer `kind`/flag branch — see the honest framing in
§2.3).

#### 2.2.8 Swallow & carry (W6 "The Gullet")
`Segment.carry?: Entity`. A `pickup` entity (flag `pickup:true`, removed from `cells` when
stepped on) is stored on the head segment as `carry` instead of growing length. Carrying:
- moves with the segment (it's part of the snake record),
- can be **deposited** via a direction + a deposit modifier into an adjacent empty cell
  (drop a key onto a plate, a block to stand on),
- a carried `solid` block, once deposited, is a normal `cells` entity again.

This expresses "swallow a thing, carry it through, regurgitate it where needed." No new
verb beyond the deposit modifier (which piggybacks on the anchor key — they never coexist
in one room). **Spike carry before authoring W6.**

#### 2.2.9 Shed-skin decoy (W6 twist)
A decoy is just **deposit of a body-shaped `solid` entity from carry**: you "shed" a held
block that then acts as static structure (a fake body / a plate-holder / a step). No new
mechanism — it is carry (§2.2.8) reskinned. **Matter-conservation oracle:** body length +
deposited matter is constant across a shed (§4.4). Cut candidate if W6 is already full.

#### 2.2.10 Co-op / relay (W7 "Two Bodies") — the one genuine fork, minimized
A second snake is the **only** ability the single-snake state cannot fake. Minimal model:
`bodies?: Segment[][]` (the *other* snakes; `snake` stays the active one). Turn model: the
player **controls one body at a time**; `Tab` switches the active body; gravity/mechanism
passes run over *all* bodies each turn. **Inter-body support is the multi-body case of the
transitive-grounding settle (§2.5):** a body is supported iff it transitively rests —
through a chain of bodies it sits on — on a world `supports` cell or an `anchored` segment.
External bodies bear weight; **two bodies resting only on each other with no ground/anchor
BOTH fall** (no mutual mid-air self-support). The settle is a simultaneous fixpoint and is
**order-independent across bodies by construction.** Win = **all** heads on (their) exits;
death = any body fully in the void. Each verb operates on the active body only.

This is ~30 lines: `settle`/`isSupported`/mechanism pass iterate `[snake, ...bodies]`. It
is the only change that touches the *root* of the state tree, so it is done **last** and
guarded by a **golden-trace migration oracle** (graft from draft-testability, §4.9): an
Inc-0 single-snake golden trace, replayed through the refactored kernel as
`bodies=[]`/active-snake-only, must produce **byte-identical** snapshots — proving the
wrap-the-kernel refactor is behaviour-preserving. **Spike co-op** (B strikes off A's back
to an exit) on one throwaway room before authoring W7.

**It is the first thing to cut** if production time is short (D-OUT-5) — W7 can be replaced
by an extra finale-style recombination world using only W1-W6 abilities, with no model
change. Recommendation: **build W1-W6 + Finale first; treat W7 as a stretch world.**

### 2.3 The "no on-ramps" ledger (the reviewer-checkable contract)

Graft from draft-testability §2.3: a table is the contract. A reviewer can verify that no
increment's diff adds a field from a later row. The model can *express* everything, but
the code carries only what the shipped world demands.

**Honest cost framing.** The flag-set's win is that the **existing rule call-sites
(step/support/eat/win) never branch per ability** — they read flags. A *new* entity is
**not free**: its "real cost" column below is the truth. Each row costs at least one
`PRESETS` row + one renderer `kind` branch; an *active* behaviour additionally costs a
resolution pass or a verb.

| Model change | Deferred until | Trigger ability | Real cost when it lands | Until then, it does not exist |
|---|---|---|---|---|
| `grip` flag + `Segment.anchored` + `anchor` verb | Inc 2 | anchor/climb | PRESETS row + renderer branch + **1 verb** + `settle` reads `anchored` as a grounding source | `isSupported` = transitive grounding on world supports only |
| `surrounds` predicate (+ optional `crushable`) | Inc 2 | wrap/constrict | one pure predicate (+ optional PRESETS row); no rule-site branch | no constriction; predicate absent |
| `trigger`/`door` flags + `triggers` set + `applyMechanisms` | Inc 3 | rattle/pressure | 2 PRESETS rows + 2 renderer branches + **1 resolution pass** | no plates, gates, or mechanism pass |
| shared `resolve` tail extracted | Inc 3 | first >1 post-action recompute | a refactor (T-RESOLVE-EQUIV differential oracle gates it) | tail is `settle(checkWin(...))` inline |
| `heat` flag + renderer fog mask | Inc 3 | dark | 1 PRESETS row + **1 renderer branch** (no rule change) | full visibility; no heat cells |
| `pickup`/`carry` + deposit modifier | Inc 4 | gullet | PRESETS row + renderer branch + deposit modifier + carry handling in step | no carry slot |
| body-shaped deposit (decoy) | Inc 4 | shed-skin | reuses carry (no new pass) | no decoys |
| `egg` flag (scoring marker) | Inc 4 | hidden eggs | 1 PRESETS row + 1 renderer branch (scoring reads the trace, outside core) | no egg cells |
| `bodies[]` (root container widening) | Inc 4 | co-op | root-state widening + multi-body iteration in settle/mechanisms/checkEnd; **mutual-support-falls** is an oracle here | single `snake: Segment[]` |

**`strikeRange` stays a plain level scalar throughout** (coil-to-charge cut, D-OUT-3) —
it never becomes derived.

**Settle oracle carried on the `bodies[]` row now:** the multi-body settle must satisfy
the *mutual-support-falls* oracle — A-on-B-on-ground holds; A-on-B-on-A (mutual,
ungrounded) both fall; the settle result is identical under any body ordering (§2.5, §4.4).

### 2.4 Turn resolution (unified pipeline)

Every verb resolves through the **same** pipeline: the current pipeline plus a shared tail
extracted at Inc 3 (the first increment with more than one post-action recompute). Each
added pass no-ops until its world exists, so Inc 0/1/2 keep their exact behavior.

```
verb(state, dir) ->
  if status != "play": return state (same ref)            [Inc0]
  apply verb's grid steps via tryStep (reads e.solid/e.eat) [Inc0, flag-driven]
    -- checkWin runs after the verb's grid steps           [Inc0]
    -- (strike) checkWin runs on EACH strike flight step    [Inc0]
  if nothing changed: return state (same ref)              [Inc0, no-op identity]
  return resolve(candidate)

resolve(s) ->
  s = settle(s)            (transitive grounding + anchored) [Inc0; +anchor Inc2]
    -- checkWin runs on EACH fall step inside settle         [Inc0]
  s = applyMechanisms(s)   (plates + occupancy -> gates)     [Inc3; no-op if no triggers]
  return checkEnd(s)       (REDUNDANT FINAL guard, not the only win check) [Inc0; multi-head Inc4]

// Win-during-fall and win-during-strike-flight are RETAINED: checkWin runs (a) after the
// verb's grid steps, (b) on each fall step inside settle, and (c) on each strike flight
// step. checkEnd is a final redundant guard, NOT the only win check.
// Inc4 (co-op): every step maps over [snake, ...bodies]; active snake acts, others
//               are external obstacles/support; win = all bodies on exits.
```

Purity is preserved end-to-end (every pass is `s -> s`), so **undo stays a snapshot stack
and no-op detection stays referential** (`next === state`). This is load-bearing and
non-negotiable.

> **Win-check semantics (corrected).** It is **not** true that `resolve == settle(checkWin(s))`
> with a single final win check. Win is detected at the **instant** it occurs: a head that
> falls through an exit cell mid-settle, or crosses an exit mid-strike, wins then and there.
> `checkEnd` at the tail is a redundant final guard, retained for safety, not the sole win
> point. Oracles T-WIN-FALL and T-WIN-STRIKE-CROSS pin this (§4.1).

### 2.5 Gravity / support — transitive grounding as a simultaneous fixpoint

Settle is defined **generally and once**, covering single- and multi-body identically. A
segment/body is **supported** iff it **transitively rests** — through a chain of bodies it
sits on — on a world `supports` cell OR an `anchored` segment. This is a *closure*, not a
one-step neighbour check.

**Algorithm (order-independent by construction).**
1. Compute the **grounded set** as the transitive closure from the grounding sources
   (world `supports` cells + `anchored` segments): a body is grounded if it sits directly on
   a grounding source, or sits on another body that is (transitively) grounded.
2. Every body **not** in the grounded set drops by 1 cell **SIMULTANEOUSLY**.
3. Repeat to a **fixpoint** (no body is ungrounded, or it has crossed `floorY`).

- **Single-body settle is the degenerate case** of this one fixpoint definition (the
  closure has one body; it is grounded iff some segment sits on a world `supports` cell or
  is `anchored`).
- **Consequence (the bug this fixes):** two bodies resting only on **each other** with no
  ground/anchor are **not** in the grounded closure, so **BOTH fall**. There is no circular
  mid-air self-support, and the result does not depend on which body is processed first
  (F3 and F17 were the same defect — order-dependence and circular support — both removed by
  the closure definition).
- **Default (W1-W2, W4-W6):** the snake is one rigid body; grounded iff it transitively
  rests on a `supports` cell. Falls uniformly toward -y; dead when *every* segment is below
  `floorY` (partial-void survives — oracle T-VOID-PARTIAL).
- **Anchor (W3+):** an `anchored` segment is a grounding source, exactly like a world
  `supports` cell — it pins the body. No bridges, no per-segment weight, no chain physics.
  The anchor is an explicit grip on a *world* surface.
- **Multi-body (W7):** the same single fixpoint runs over `[snake, ...bodies]`. A body may
  rest on another body, but the chain must terminate at a real grounding source — mutual
  ungrounded pairs fall.

**Termination (P-SETTLE-TERMINATION still holds):** each ungrounded component strictly
descends by 1 per iteration until it is grounded or has crossed `floorY`, so the fixpoint is
reached within `(maxY - floorY + 1)` iterations. A hard iteration cap inside `settle` asserts
the bound is never exceeded (§4.0, §7-note in §4.0).

**Oracles (added):** A-on-B-on-ground holds; A-on-B-on-A (mutual, ungrounded) both fall;
the settle result is **identical under any body ordering** (§4.4). The mutual-support-falls
oracle is carried on the `bodies[]` ledger row now (§2.3).

### 2.6 World / level curriculum

Eight worlds, each: **introduce** (clean teach) -> **twist** (break the assumption) ->
**recombine** (finale that mixes the world's ability with prior ones). Target ~6
rooms/world, **~48 rooms** as a *ceiling, not a quota* — enough to teach the toolbox
without a content death-march. Per-room author cost is the real content risk (§5). A world
never introduces two new mechanics. Every room is a readable vertical plane.

**Authoring ergonomics land in Inc 1, not later.** Rooms are **drawn, not coordinate-typed**:

- **ASCII-art room format.** A room is rows of characters parsed into cells by a **pure,
  unit-tested parser** (e.g. `#`=wall, `o`=fruit, `X`=exit, `S`=snake, `.`=air; later
  worlds add glyphs for `grip`/`plate`/`gate`/`pickup`/`egg`). The parser produces a
  `LevelDef`; it is a pure function with a unit-tested oracle (§4.1, T-ASCII). **This is
  introduced before room ~5** (not "after ~40 rooms" — see D-OUT-2).
- **Record-par-from-undo.** A one-key shell action records the current play session as the
  room's par solution, **reusing the existing undo history** (the move sequence is already
  the snapshot stack). The author solves the room once, presses the key, and the recorded
  optimal feeds T-PAR (§4.5).

**Content ownership is an explicit OPEN success-criterion (§5, §6 D23):** *who* authors the
~48 rooms, and whether one person can realistically design, solve, par, and playtest them.
The ASCII format + record-par tool exist precisely to make that load bearable; it remains an
open question to resolve before committing to the full curriculum.

### 2.7 Scoring model (eggs + move-par) — optional, non-blocking, outside the core

Eggs/par/constraints live **outside the pure core**, as functions over the snapshot trace
the shell already keeps for undo — zero new core state, zero undo risk. The core answers
"is this move legal and what's the new world"; nothing about score.

- **Eggs (stars), max 3/room, all optional:** (1) **solve** — `status==="won"`;
  (2) **hidden** — touch a marked `egg` entity en route (collected like fruit but tracked,
  not grown; the flag is derived from the trace); (3) **constraint** — solve under a
  per-room predicate over the trace (`maxLength`, `maxMoves`, "no strike"). Eggs **loosely
  gate**: a world unlocks at a threshold the *solve* eggs alone exceed, so a player who
  only solves rooms always progresses (eggs gate *bonus* content, never the main path). A
  skip-after-K-failed-attempts path guarantees nobody is hard-walled.
- **Move-par:** the **hand-authored** optimal move count per room, stored in
  `LevelDef.par`, **shown only after first solve**. No solver is built (a solver is a
  project unto itself; cut). Par correctness is verified by an authoring oracle (T-PAR): a
  recorded optimal solution (captured by the record-par tool, §2.6) replayed by the test
  harness reaches `won` in exactly `par` moves — over *every* shipped room. **Caveat
  (honesty note):** T-PAR proves par is **achievable**, not provably **optimal** — par
  optimality is **author-asserted** (no solver verifies minimality). See §4.5, §6 D15.
- **Persistence:** one `localStorage` key (`coil.progress.v1`) holding highest world +
  egg set. Read/written through a tiny **injected store** so the serialize/parse
  round-trip is unit-tested off the real `window` (graft; §4.5). Move counter + egg
  progress live in `main.ts` (derived from `history.length`), **not in `GameState`** — so
  undo naturally rewinds them and they never pollute the pure core.

### 2.8 Renderer / animation approach

- **Clear-and-rebuild is the firm, permanent renderer decision.** Rooms are tiny; the
  current `render(state)` (dispose + rebuild meshes) is correct, simple, and already works.
  It is the **only firm renderer decision and the permanent fallback.** **No animation in
  Inc 1** — discrete snap-to-grid reads fine for a turn-based puzzle, and the slice's job is
  to prove the *feel of the rules*. Inc 1 adds only **camera auto-fit** (compute scene
  bounds from `cells`+`snake`, set `controls.target`/distance) so rooms beyond level1's
  range frame correctly (resolves the hardcoded `(6,2,0)`).
- **Animation is not committed.** *If/when* animation is later added (the first increment
  where multi-cell strike + mechanism reads are genuinely hard to follow as instantaneous
  redraws — practically Inc 2/3), it would use **persistent meshes** keyed by identity and
  **carve the snapshot-diff into a pure, tested function** at that point — but no
  `diffPlan.ts` file and no `T-DIFFPLAN` oracle are committed up front (that was
  over-engineering for a deferred, maybe-never feature). The renderer can **always fall back
  to a full clear-and-rebuild** on any diff it cannot plan — the permanent safety valve.
- **Strike trace (render-only, if animation lands):** `move`/`strike` could optionally
  return `{state, trace}` where `trace` is the ordered intermediate head cells,
  **render-only metadata that never enters the snapshot or the undo stack** (graft from
  draft-risk §2.4). `main.ts` would feed `trace` to the renderer and store only `state` in
  history; undo never sees a trace. This is a fallback if curved/interrupted strikes ever
  make renderer-side path reconstruction ambiguous — not built for Inc 1.
- **Any future animation work is purely additive to the renderer** — it touches no other
  layer and does not gate Inc 1. If it gets hairy, clear-and-rebuild is a permanently
  acceptable fallback.
- **Entity rendering generalizes trivially:** the renderer switches on `entity.kind` for
  mesh/color and reads `heat` for the Dark lighting mode. No rules leak in.
- **Snake look:** Inc 1 keeps boxes (head distinct). A tube/oriented-segment look is a
  nice-to-have slotted with any future animation work, not a blocker.

## 4. Acceptance criteria / test oracles (definition of done)

The test plan **drives the model**: a feature is not "done" until its oracle is written
(red) then green. All core oracles are **vitest** (node env), pure `state -> assert`,
deterministic (no clock; randomness only via the seeded generator below, §4.0), run with
stdin closed. Renderer/almanac oracles are partly **scripted** (the two hard ship gates,
§2.9) and partly a **manual checklist** for the irreducibly visual checks.
**Definition of done for an increment = all its oracles pass + `tsc --noEmit` clean.**

### 4.0 Core-wide property battery (graft from draft-testability §4.0) — from Inc 1, forever

These invariant oracles are re-run against **every verb** — the cheapest, highest-leverage
testability multiplier. They run against **one seeded, deterministic randomized GameState
generator**:

- **The generator.** A single `fast-check` `Arbitrary<GameState>` over small grids: random
  snake placement, random preset cells, random `strikeRange`/`floorY`. The battery runs
  against a **fixed seed corpus of N cases** plus **pinned regression seeds** (seeds that
  once found a bug are kept forever). The generator **grows** to cover `carry` / active
  `body` when those land. **`fast-check` is added as a TEST-ONLY devDependency**
  (proposed new dependency).
- **Verb registration is the gate.** Every verb (including future verbs) MUST be
  **registered** with the battery; the registration list is what the P-* properties iterate.
  Adding a verb without registering it is a visible omission — the registration IS the gate.

Properties:

- **P-DETERMINISM:** `v(s, dir)` is byte-identical (deep-equal) on repeated calls, over the
  generated corpus.
- **P-PURITY:** after `v(s, dir)`, `s` deep-equals a pre-call clone (no mutation).
- **P-NOOP-IDENTITY:** a blocked/terminal action returns the **same reference** (`toBe`),
  so the shell's `next === state` no-op detection holds.
- **P-UNDO:** apply `v` then restore the previous snapshot deep-equals the original, for
  every registered verb (the snapshot stack is the undo spec). Extended to non-null `carry`
  and to an `active`-body switch as those land.
- **P-CONSERVATION (scoped to what exists at each increment):** at **Inc 1**, every
  non-eating action leaves `snake` length unchanged and an eating action is exactly +1.
  **Extended at Inc 4** to carry/shed per the ledger (a shed conserves body + deposited
  matter; carry adds 0 to length).

**Settle termination is a DIRECTED test, not a fuzz property (scope decision, see C7/D-note):**

- **P-SETTLE-TERMINATION (directed):** a high snake over a deep void reaches `dead` in the
  expected iteration count (an exact directed assertion), **PLUS** a **hard iteration-cap
  assertion inside `settle`** that fires if the `(maxY - floorY + 1)` bound is ever exceeded.
  Property-fuzzing of settle is **deferred** until a mechanic makes settle non-trivial. (The
  seeded generator above still serves the other P-* properties.)

- **T-STRIKE-4** strike *through a fruit* eats mid-flight: length+1, fruit removed, flight

- **T-STRIKE-5** strike that hits a wall stops short (lands at last legal cell).

  regression:** all prior tests pass byte-identically with heat cells added.

### 4.4 Increment 4 — carry (W6) / decoy / co-op (W7)
- **T-CARRY-1** stepping onto a `pickup` stores it as `carry` (length unchanged, cell
  cleared); swallow with full gut -> no-op (same ref).
- **T-CARRY-2** depositing places the entity in the target empty cell as a normal entity;
  a deposited `solid` then blocks/supports; deposit into a blocked cell -> no-op (same ref).
- **T-DECOY** a shed block acts as static structure (supports / holds a plate);
  matter-conservation holds.
- **T-SETTLE-MUTUAL** *(transitive grounding oracles)* A-on-B-on-ground **holds**;
  A-on-B-on-A (mutual, ungrounded) **both fall**; the settle result is **identical under
  any body ordering** (order-independence by construction).
- **T-COOP-MIGRATE** *(the migration gate)* an Inc-0 single-snake golden trace replayed
  through the multi-body kernel (`bodies=[]`) yields **byte-identical** snapshots.
- **T-COOP-1** *(if W7 shipped)* switching bodies moves the active pointer; verbs affect
  only the active body; one body supports the other (transitive grounding), each settles
  in the single simultaneous fixpoint.
- **T-COOP-2** *(if W7 shipped)* win requires **all** heads on exits; any body fully in
  void -> dead; settle/mechanisms run over all bodies.
- **Spikes I4a (carry) and I4b (co-op) green before authoring W6/W7.**

### 4.5 Scoring oracles
- **T-PAR** for *every* room, a recorded optimal solution (captured by the record-par tool,
  §2.6) replayed reaches `won` in exactly `LevelDef.par` moves (catches a wrong/now-
  impossible par after any engine change). **Caveat:** this proves par is **achievable**,
  not provably **optimal** — par optimality is **author-asserted** (no solver is built).
- **T-EGG-HIDDEN** a scripted solution touching the marked `egg` flags it; one avoiding it
  does not.
- **T-EGG-CONSTRAINT** the constraint predicate is true for a compliant solve, false for a
  violating one.
- **T-GATING** the world-unlock threshold is reachable by *solve eggs alone* (no main-path
  wall); the skip-after-K path also unlocks.
- **T-PERSIST** the `coil.progress.v1` serialize/parse round-trip is unit-tested through
  the injected store (off the real `window`).

### 4.6 Renderer oracles
- **M-RENDER (manual)** the renderer constructs without throwing for a tiny state; the
  snake visibly slithers with no flicker; orbit works; the full clear-and-rebuild fallback
  engages cleanly on an unplannable diff. *(Irreducibly visual — stays manual.)*
- *(No committed `diffPlan` / `T-DIFFPLAN`: if/when animation is added, carve the
  snapshot-diff into a pure tested function at that time — §2.8.)*

### 4.7 Almanac ship oracle (scripted + manual)
**Scripted (postbuild in `mdsnake`, fail the build):**
- **No absolute `/assets/`** — postbuild greps `dist/index.html`; any absolute `/assets/`
  reference fails the build. (Automates the one hard ship constraint; was M-SUBPATH.)
- **M-BUNDLE** — postbuild asserts gzipped `assets/index-*.js` is within ~1.5x night-cab's;
  else fails.
- **M-CORE** — `vitest run` and `tsc --noEmit` clean in the source repo before copying
  `dist/` (CI-style scripted check, run before ship).

**Manual checklist (every ship):**
- **M-LOAD** `http://localhost:8000/coil/` loads with no console error; the canvas renders
  Room 1; the game is playable.
- **M-LISTING** the new `ESSAYS` entry renders a card in `#listing` (and `about.html`
  stats), with the chosen tag chip filtering it and the `ill-coil` icon (not fallback).
  *(No almanac test harness exists — this is the only truly manual ship check.)*

## 5. Residual risks

1. **Entity refactor regresses the slice.** Mitigated by **ordering**: the §4.0 property
   battery + §4.1 oracle-gap tests are landed GREEN on the **pre-refactor** code FIRST, then
   the refactor is done against that hardened net (the refactor is **not** the first Inc-1
   commit — the hardening is). The refactor is mechanical (presets reproduce the three types
   exactly) and the existing + new tests are the oracle; because they were proven green on
   the old code, any post-refactor red is unambiguously the refactor's fault. T-RESOLVE-EQUIV
   adds a differential check for the tail extraction.
2. **Animation scope creep — pre-empted.** No animation and no `diffPlan` module are
   committed; clear-and-rebuild is the firm, permanent decision and fallback. If animation is
   ever added it is renderer-isolated and carves out a pure tested diff function at that time
   (§2.8). The risk is deferred to a maybe-never feature, not carried now.
3. **Co-op (W7) cost.** Even minimal, multi-body touches the state root, settle/mechanism
   passes, and input. Mitigated: it is **cut-first**; done **last**; the settle is the same
   transitive-grounding fixpoint (order-independent, mutual-support-falls), and it is guarded
   by the golden-trace migration oracle (T-COOP-MIGRATE) and a spike. The game is complete
   without it.
4. **Move-par authoring burden.** Hand-authored par needs a recorded optimal per room —
   captured cheaply by the record-par-from-undo tool (§2.6). Mitigated by T-PAR (a wrong par
   fails the build) and showing par only post-solve. T-PAR proves *achievable*, not
   *optimal* (author-asserted). No solver is built (YAGNI at ~48 rooms).
5. **`crushable`/constrict (§2.2.5) may be a feature nobody needs.** Flagged as a cut
   candidate; build only if a W2 twist genuinely requires it, else delete the predicate.
   The flood-fill definition removes the geometric ambiguity if it does ship.
6. **Content production is the real long pole**, not engineering. ~48 hand-designed,
   solvable, fun rooms is the bulk of the work; the model is small by design precisely so
   effort goes here. **Content ownership is an explicit OPEN success-criterion** (§2.6):
   *who* authors the rooms and whether one person can realistically do it. Mitigated by the
   ASCII room format + record-par tool landing in Inc 1, per-world increments (value lands
   continuously), spike-before-content for structural worlds, and T-PAR catching broken
   authoring. **"Fun"/"difficulty" are validated by human PLAYTEST, not an oracle**; T-ROOM-
   SOLVE ("solvable") is the machine-checkable floor.
7. **Bundle size (`three` + OrbitControls).** Unmeasured; same risk night-cab already
   accepted. Mitigate by the **scripted** M-BUNDLE gzip-budget gate at Inc 1, importing only
   used `three` modules, matching night-cab's `three` version; not expected to block a Pages
   doc.
8. **Anchor as a third control.** Slightly grows the surface. Mitigated: it's a *toggle*,
   introduced only in W3 with a dedicated teach room; two movement verbs + one toggle still
   fits in the head.
9. **Darkness-as-render could disappoint** if players expect fog to be a "rule."
   Mitigated: it's a deliberate testability trade; the puzzle (limited-info navigation) is
   unaffected; reversible later (it would become a real model change with its own oracle).
10. **Two-repo mirror drift.** `coil/` in the almanac is build-output-only and never edited
    there; the recorded source SHA (`coil/SOURCE.txt`) makes any almanac<->mdsnake HEAD
    divergence detectable. A single-repo alternative was considered and rejected (D3); the
    mirror discipline + SHA check is the mitigation.
11. **New test dependency (`fast-check`).** Proposed TEST-ONLY devDependency for the seeded
    property generator. Mitigated: test-only (never ships in the bundle), and the generator
    is the one place randomness enters the otherwise-deterministic test suite.

A gate is held open by **EITHER its plate OR a body in its mouth**, derived every turn from current triggers + occupancy with **NO cross-turn latched state**; occupancy is an existential test over ANY segment (order-independent). Keeps the theme (your body triggers the world) with one pure `applyMechanisms` pass. Preserves purity/undo/determinism. |
| **D10** | **Dark/heat-sense is renderer-only; zero core *rule* change.** | The cheapest world: same rules, a lighting mode reading `heat` (still one renderer `kind`/flag branch — not literally free). Animation is a maybe-never deferred feature — if added, carve the snapshot-diff into a pure tested function at that time. Never gates the first release. |
| **D14** | **Score-light; eggs/par/constraints live OUTSIDE the core as trace functions; one `localStorage` key via an injected, round-trip-tested store.** | Keeps the tested surface tiny; scoring can't corrupt rules or undo. A stuck player still advances (loose gating + skip-after-K). |
| **D15** | **Move-par hand-authored + verified ACHIEVABLE by replay (T-PAR) over every room; no solver.** Par optimality is **author-asserted**, not machine-proven. | A solver is a project unto itself (cut). Replaying a recorded optimal catches a wrong/now-impossible par cheaply; minimality is the author's claim, stated honestly. |
| **D16** | **Install a core-wide property battery (determinism, purity, no-op identity, undo round-trip, conservation) from Inc 1, re-run against every registered verb.** Settle-termination is a directed test + a hard iteration cap, not a fuzz property (yet). | The cheapest, highest-leverage testability multiplier; cross-cutting invariants that per-mechanic directed tests miss. Settle is trivial until a mechanic complicates it, so a directed test + cap suffices. |
| **D17** | **Spike-before-content for the three structural extensions (anchor, carry, co-op).** | Validate the feel on one throwaway room before authoring a world, so a bad mechanic costs a spike, not a world of content. |
| **D18** | **Increment-sequenced delivery; Inc 1 (World 1) ships standalone with zero new model surface.** | Each increment is independently buildable/testable/shippable; value lands continuously and the model grows only as worlds demand. |

| **D20** | **Multi-body settle = a simultaneous fixpoint over transitive grounding.** A body is supported iff it transitively rests (through a chain of bodies) on a world `supports` cell or an `anchored` segment; all ungrounded bodies drop by 1 simultaneously; repeat to fixpoint. | Fixes circular mid-air support AND order-dependence (F3 and F17 are the same defect). | **D21** | **Land the property battery + oracle-gap tests GREEN on the pre-refactor code FIRST, then do the entity refactor against that hardened net.** | Breaks the circular "tests pass => behaviour preserved" reasoning: the net is proven green on known-good code first, so any later red is the refactor's fault. The parser oracle (T-ASCII) and record-par reuse of the undo history make per-room authoring cheap; honesty about who authors and how fun is judged. |
